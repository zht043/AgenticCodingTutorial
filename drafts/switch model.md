太棒了！在日常开发中，掌握“看菜下饭”的模型切换策略，不仅能让你的开发效率起飞，更能帮你省下大把的 API 费用（Opus 的价格通常是 Sonnet 的好几倍）。

在 Claude Code 这种终端 Agent 中，你可以把 Sonnet 4.6 想象成“极其敏捷的高级开发工程师”，把 Opus 4.6 想象成“深思熟虑的首席架构师”。

以下是具体的切换配置方法与实战工作流：

1. 怎么在 Claude Code 中随时切换模型？
在 Claude Code 的交互式终端（CLI）中，你不需要修改复杂的配置文件，可以通过内置的**斜杠命令（Slash Commands）**实现热切换：

查看/切换当前模型： 在对话框中直接输入 /model 并回车。终端通常会弹出一个交互式列表，你可以直接用方向键在 Sonnet 和 Opus 之间切换。

启动时指定模型： 如果你一上来就知道这个任务极其复杂，可以在启动命令行时直接加上参数（具体名称视官方标志而定），例如：claude --model claude-4.6-opus。

2. “高低搭配”的实战策略（什么时候用谁？）
🚀 场景一：使用 Sonnet 4.6（承担 80% 的日常工作）

作为你的默认首选模型。

适用任务：

编写常规的业务逻辑（增删改查 CRUD）。

补充单元测试（Unit Tests）或注释。

修复报错信息非常明确的 Bug（例如类型不匹配、空指针异常）。

格式化代码或进行简单的重构。

核心优势： 快与省。 生成代码的速度极快，不会让你在终端前干等；同时 Token 成本低，让 Agent 在本地大量执行文件读取（read_file）和搜索（grep）时，你不至于肉疼。

🧠 场景二：切换到 Opus 4.6（承担 20% 的硬核攻坚）

作为你的兜底专家和架构师。

适用任务：

跨文件的全局重构： 需要同时修改底层数据模型、中间件和前端接口，牵一发而动全身。

从零设计新系统： 需要搭建高并发队列、设计复杂的鉴权机制或数据库表结构。

解决“玄学 Bug”： Sonnet 尝试了两次都没修复成功，或者涉及多线程死锁、内存泄漏等需要深度逻辑推理的问题。

核心优势： 深与稳。 它能 hold 住极其庞大的上下文，逻辑推理链条极深，不会在复杂的代码依赖中“迷路”。

3. 高手进阶：压榨性价比的“黄金连招”
真实开发中最最高效的组合拳是这样的：

遇事先让 Sonnet 上： 把任务交给 Sonnet 4.6，让它去读取本地文件并尝试修改。

陷入僵局时及时打断： 如果 Sonnet 连续跑了两次测试都报错，或者它开始在同一个文件里来回修改（陷入了 Agent 的“死循环”），立刻按 Ctrl+C 打断它。

切换 Opus 终结比赛： * 先用 /compact 或 /clear 命令稍微清理一下前面无关的试错聊天记录（极其重要，因为 Opus 按 Token 计费，不要把废话传给它）。

输入 /model 切换到 Opus。

告诉它：“刚才的思路行不通，报错依然是 [Error Log]，请重新审视 X 模块和 Y 模块的依赖关系，给出根本的解决方案。”

通过这种模式，你既享受了 Sonnet 的速度和低价，又拥有了 Opus 的兜底能力。
