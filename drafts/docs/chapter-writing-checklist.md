# Agentic Coding 教程单章写作检查清单（草案 v0.1）

> 用途：在扩写或重写任意一章之前/之后，用这份清单快速自查，保证整本书的风格、结构和信息密度保持一致。

---

## 一、结构层面检查

1. 是否明确回答了这 6 个问题？
   - 这一章解决什么问题？  
   - 读者为什么需要它？  
   - 核心概念或核心流程是什么？  
   - 是否有一段“最短可执行示例”？  
   - 常见误区或边界是什么？  
   - 本章实践清单或落地建议是什么？  

2. 章节标题是否尽量“问题导向 / 动作导向”，而不是堆概念？  
   - 例如：  
     - ✅「计划先行：让 Agent 先分析，再执行」  
     - ❌「关于 Agent 的一些思考」  

3. 是否有清晰的小节层级（最多 2 级）？  
   - 不要出现 3 级以上小标题嵌套。  

---

## 二、内容密度与示例

4. 是否区分了“理论段落”和“实操段落”？  
   - 理论段落：解释概念和原则，尽量短；  
   - 实操段落：给出 prompt 模板、命令示例、文件示例、伪代码或流程图。  

5. 本章是否至少包含一个“最小可运行示例”？  
   - 可以是：  
     - 一段可直接复制使用的 prompt；  
     - 一个 shell 命令 + 预期输出说明；  
     - 一个最小目录结构示意；  
     - 一小段伪代码或 YAML 配置。  

6. 是否避免“空心建议”？  
   - 如：只写“要注意安全”“要注意上下文污染”，但没有配任何可执行的做法。  

7. 示例是否与本章主题高度贴合？  
   - 避免用过于复杂或与主题无关的示例。  

---

## 三、与全书主线的一致性

8. 是否明确承接前文、指向后文？  
   - 例如在 Part 2 提到的闭环（Goal → Plan → Act → Observe → Verify → Iterate），在后续章节里是否按同一套说法引用。  

9. 是否在适当位置引用了“所有 Agent 都是 Coding Agent”的视角？  
   - 尤其是在讲多 Agent、垂直 Agent、行业场景时。  

10. 是否与 Part 4 的 Prompt / Context / Verification 三主梁保持一致？  
    - 本章在描述方法论时，是否能明确标出来主要落在这三者中的哪一根。  

---

## 四、外部资料与版本敏感内容

11. 是否使用了 `docs/research-sources.zh-CN.md` 中推荐的官方资料作为论据？  
    - 优先引用官方文档、官方博客、官方仓库；  
    - 社区文章和视频作为补充，而不是主线依据。  

12. 是否避免在正文中写死以下信息？  
    - 具体模型版本号、参数规模；  
    - 价格、费率组、限速细节；  
    - 某个兼容层 / 中转服务是否“当前可用”；  
    - 某个插件、工具的维护状态。  
    - 若必须提及，是否记录在 `docs/version-sensitive-matrix.md` 或附录中？  

13. 是否在需要时添加“进一步阅读 / 推荐资料”小节？  
    - 只列 2–5 条最关键的官方/高质链接；  
    - 与正文内容强相关，避免泛泛罗列。  

---

## 五、风格与风险提示

14. 表述是否尽量“工程化”，避免空泛形容词？  
    - 多用“应该/必须/建议 + 具体动作”，少用“很重要/很强/很牛”等评语。  

15. 是否在高风险话题处加入了清晰的风险提示？  
    - 如权限、安全、生产环境操作、第三方中转服务。  

16. 是否清楚地说明了“人类在环”的位置？  
    - 哪些地方必须人工确认；  
    - 哪些地方可以全自动；  
    - 谁对结果负责。  

---

## 六、TODO 与后续扩展位

17. 本章是否明确标注了 TODO 位，而不是隐性“以后再说”？  
    - 如：`TODO：补一张三层行动空间示意图`；  
    - `TODO：配一个最小 Ralph Wiggum 循环伪代码示例`。  

18. TODO 是否控制在可管理数量？  
    - 每章不宜超过 3–5 个大 TODO；  
    - 每个 TODO 尽量具体到“补什么类型的内容”。  

---

## 七、适用于本书的快速自评等级（可选）

写完一章后，可以用 1–3 级给自己一个粗略评分，方便后续迭代排序：

- 1 级：结构齐全，至少一个可运行示例，没有明显概念错误，可直接面向读者；  
- 2 级：主线清楚，但示例/插图不够，TODO 较多，适合内部预览；  
- 3 级：思路草案，缺示例、缺结构，主要记录想法，不宜直接发布。  

> 建议：优先把全书所有章节都提升到 2 级，再按读者需求和精力重点把 Part 1、Part 4、Part 5、Part 8 等关键章节打磨到 1 级。
