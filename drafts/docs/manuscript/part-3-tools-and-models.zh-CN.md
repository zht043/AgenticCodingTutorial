# Part 3 工具与模型地图

写 Agentic Coding 教程，最容易滑向两个极端。一个极端是做成“产品推广合集”，每个工具都说一点，最后读者什么都没记住。另一个极端是完全不比较，只讲抽象方法论，结果读者不知道该怎么选。更好的方式是：先把产品和模型分层，再谈差异。

## Chapter 10 主流 Code Agent 全景图

先说最重要的一句：工具名不等于能力本质。你真正要比较的是“它把模型放进了怎样的任务闭环里”。

下面是一张适合放进正文的简化视图：

| 工具 | 主要形态 | 强项 | 风险或边界 | 更适合谁 |
| --- | --- | --- | --- | --- |
| `Claude Code` | 终端 / IDE 集成 | 终端工作流强，任务闭环清晰，适合工程主线 | 更依赖命令行习惯 | 想真正学会 Agent 方法的人 |
| `Codex` | CLI、IDE、云端 Agent | 本地与云端形态都强，适合并行任务与平台化协作 | 产品形态较多，新手容易混淆 | OpenAI 生态用户 |
| `Cursor` | IDE 原生 Agent | 编辑体验成熟，规则体系清晰，适合平滑升级 | 容易被用成“更强 IDE Chat”而没进入真正 Agent 工作流 | IDE 重度用户 |
| `Cline` | IDE 插件 / CLI | 审批流程直观，MCP 扩展强，兼容 provider 多 | 配置空间大，也更容易配乱 | 想自己掌控接入层的人 |
| `GitHub Copilot coding agent` | GitHub 平台 / 云端执行 | 与 issue、PR、审查流程天然整合 | 更偏平台型任务，不是所有本地探索都适合 | 团队、平台协作场景 |
| `Trae` 等区域性工具 | IDE / 平台混合 | 本地生态、中文体验、集成便利性可能更友好 | 长期能力和生态要具体评估 | 区域性生态用户 |

### 怎么理解这些差异

`Claude Code` 更像“终端中的工程代理”，非常适合教学，因为它把核心动作都暴露得很清楚。你能明显看到它什么时候在读文件、什么时候在计划、什么时候在跑命令、什么时候在等待权限。

`Codex` 的价值在于它把本地 CLI、IDE 接入和云端 Agent 连接起来。这样你不仅能理解本地开发辅助，还能理解“把任务派发到隔离环境并行执行”的产品方向。

`Cursor` 则非常适合作为“从 IDE 助手向 Agent 迁移”的中间站。很多开发者本来就离不开编辑器，Cursor 的优势在于它让规则、上下文和代码修改体验高度融合。

`Cline` 的独特价值是把“人机协同边界”做得很显性。每一步权限审批、终端执行、浏览器操作都更可见。对于理解 Agent 的安全边界和扩展能力，这很有帮助。

`GitHub Copilot coding agent` 更像一个平台型 Agent。它的强点不是“在你本地终端里跟你一起敲”，而是“在 GitHub 的工作流里接 issue、开 PR、接受 review”。这特别适合团队流程，而不只是个人写代码。

### 不要做的比较

不要只比较“谁更聪明”。更实用的比较维度是：

- 它主要在本地干活，还是在云端干活？
- 它的权限模型是默认放权，还是默认审批？
- 它更适合探索、实现、review，还是平台流转？
- 它是否支持规则文件、记忆、MCP、脚本化扩展？
- 它在你的真实工作流里，到底能不能跑完整个任务闭环？

## Chapter 11 主流 Code LLM 图谱

这一章必须反复强调：模型不是 Agent。模型负责推理和生成，但它不天然等于一个完整的工作流系统。

站在 2026 年初这个时间点看，写教程时可以把代码相关模型大致分成几类：

### `Claude` 系列

整体印象通常是长任务稳定性、代码理解、工具协作和工程风格较强。很多终端型和 Agent 型产品也更愿意围绕这类模型构建工作流。

### `GPT / Codex` 系列

整体优势在于产品线完整，既有模型能力，也有 CLI、IDE、云 Agent、SDK 等形态，适合把“模型能力”和“平台能力”打包考虑。

### `Gemini` 系列

多模态和云生态整合能力常常是一个优势，尤其适合和 Google 生态、文档、设计稿、云服务打通的场景。

### `Kimi`、`GLM` 等中文开发者常见模型

这类模型对于中文语境、成本、区域可达性和兼容接入往往更有现实意义。对于中文教程来说，不能忽视它们，因为很多读者真正会用到的不是抽象 benchmark 第一名，而是“我现在能稳定接上且成本能接受的模型”。

### 怎么看待“版本对比”

很多读者都喜欢直接问：“哪个最强？Opus 对 Codex 谁赢？某个版本对某个版本呢？”这种问题不是不能答，但不应该成为正文主线。因为版本更替很快，今天的答案很可能下个月就不稳了。

更推荐的写法是教读者如何自己评估：

1. 选 3 到 5 个真实任务。
2. 统一仓库、统一规则、统一验收标准。
3. 观察计划质量、实现质量、验证行为、恢复能力和成本。
4. 最后再看模型名。

### 真正影响体验的乘积模型

一个更准确的公式是：

`实际体验 = 模型能力 x Agent 框架 x 上下文质量 x Prompt 结构 x 验证流程`

只盯模型名，通常会高估模型、低估工作流。

## Chapter 11.5 文件系统、CLI 与 RAG 的分工（TODO：补示意图）

很多人一开始接触 Agent，就会被各种「RAG」「向量库」「知识库」的说法吓到，以为不搭一套大规模检索系统就没法做 Agentic Coding。实际上，xhs notes 和业界不少实践都在强调另一种更朴素的划分：对绝大多数工程任务来说，文件系统和 CLI 就已经是最好的知识库和行动总线。

可以先用一句话记住：

> 对工程任务而言：文件系统是默认知识库，CLI 是默认行动总线，RAG 是针对特别大的松散知识的“专用武器”。

### 文件系统：默认的长期记忆与工作空间

在代码项目中，你最重要的知识几乎都已经在仓库里了：

- 代码本身
- 配置文件
- README / 设计文档 / 架构图链接
- 测试用例
- 迁移脚本与运维脚本

把这些内容好好组织（目录结构清晰、命名一致、有说明文件），再让 Agent 直接读写这些文件，本身就构成了一个天然的知识库，比把一切都塞进外部向量库更直观，也更容易调试。

### CLI：统一的行动总线

当 Agent 需要“做事”时，CLI 往往是最通用、最可组合的接口：

- 运行测试、build、lint、格式化；
- 启动本地服务、脚手架、代码生成工具；
- 调用当前语言生态里的各种管理命令。

与其为每个动作都写一个 RPC 工具，不如优先考虑是否可以通过标准命令行解决。这样做的好处是：

- 用人类也习惯的方式观察输出和调试；
- 更容易在 CI / 本地脚本 / Agent 之间共享同一套命令；
- 出现问题时可以直接在终端复现，而不用追踪复杂协议。

### RAG / 向量库：特大体量、弱结构知识的专用方案

当然，RAG 不是没用，而是更适合解决另一类问题：

- 大量 PDF、网页、扫描文档；
- 跨项目积累的经验知识库；
- 体量巨大、结构松散、无法直接放进单个仓库的内容。

在这些场景下，向量检��、Contextual Retrieval 等方法可以显著提升“找到相关信息”的概率。但这并不意味着日常工程任务必须先上 RAG。相反，如果你只是维护一个或几个代码仓库，那么：

- 把仓库结构整理好；
- 配好 AGENTS.md / CLAUDE.md / 规则文件；
- 让 Agent 熟练使用 `grep` / `rg` / 语言自带工具搜索；

往往比第一天就搭一套 RAG 更划算。

## Chapter 12 如何选工具、选模型、选组合

真正实用的教程，不应该把所有读者都引向同一个方案。更合理的做法是给出决策树。

### 场景一：新手，想尽快上手

首要目标不是“最便宜”，也不是“最强”，而是“最少变量”。这类读者推荐：

- 主线选一个工具，不要同时学四个。
- 优先选官方支持路径。
- 优先选能清楚看到任务闭环的产品。

对这类人，我会推荐两条路线：

- 想理解 Agent 本质：先用 `Claude Code`
- 想从现有编辑器平滑升级：先用 `Cursor`

### 场景二：个人开发者，预算有限

这类人要关注的是“把贵模型用在刀刃上”。推荐策略：

- 强模型做需求理解、架构分析、复杂定位。
- 快模型或便宜模型做批量改动、机械迁移、文档整理。
- 把验证尽量自动化，避免反复无效试错。

### 场景三：团队或企业

这类人最重要的不是单次体验，而是：

- 审计
- 权限
- 数据边界
- 成本可预测性
- 与现有 Git / PR / CI 工作流的融合

这时工具选择往往不再是“谁最爽”，而是“谁最容易被制度化、团队化”。

### 场景四：按任务类型选

可以直接这样给读者建议：

- 陌生仓库理解：优先选仓库搜索和终端闭环强的 Agent
- 复杂 bug 修复：优先选验证链强、能持续观察输出的 Agent
- 批量机械改动：优先选规则体系稳定、patch 质量高的 Agent
- 文档与知识整理：模型本身和规则文件比 Agent 外壳更关键
- Issue 到 PR：优先考虑云端或平台型 Agent

### 一个非常实用的组合思路

不要执着于“一把梭”。很多成熟工作流其实是组合式的：

- `Claude Code` 做仓库探索和实现主线
- `Codex` 做并行任务或云端委派
- `Cursor` 做细粒度代码 review 和可视化改写
- `Cline` 做多 provider 接入和 MCP 实验

只要你脑中清楚谁负责什么，这种组合反而比单工具更强。
