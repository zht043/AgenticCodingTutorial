# Part 4 方法论

这一部分是全书真正的分水岭。因为到了这里，教程不再只是在介绍产品，而是在教读者如何稳定地把 Agent 用好。很多人会把好体验归因于模型，把坏体验归因于运气；但从长期看，Prompt、Context 和 Verification 才是决定上限的三根主梁。

## Chapter 13 面向 Agent 的 Prompt 技巧

给 Chatbot 写 prompt 和给 Agent 写 prompt，不是一回事。Chatbot prompt 更像“提问艺术”，Agent prompt 更像“任务规格说明”。

一个好用的 Agent prompt，至少应该清楚这六件事：

- 背景是什么
- 目标是什么
- 约束是什么
- 允许它操作哪些范围
- 如何验证完成
- 你希望它最后以什么形式交付

一个可以贯穿全书的通用模板如下：

```text
背景：
这里是一个已有项目，技术栈是 X，当前问题是 Y。

目标：
完成 Z。

约束：
不要改动 A，不要引入 B，不要修改无关文件。

上下文：
优先查看这些文件/目录：...

工作方式：
先分析并给出计划，再执行。
如遇不确定点，先停止并说明。

验证：
完成后运行 ...，并说明验证结果。

交付：
输出改动摘要、涉及文件、验证结果、剩余风险。
```

这类 prompt 的好处在于，它把“猜你的心思”变成了“执行你的规格”。尤其对于复杂任务，结构比辞藻重要。

### Prompt 里最容易缺失的部分

很多人会写目标，却不写边界；会写想做什么，却不写怎么算完成。这会导致 Agent 在任务中途不断自我扩张，比如顺手重构、顺手升级依赖、顺手改风格，最后把一个小任务做成大工程。

### Prompt 的正确粒度

太粗，Agent 会乱跑。太细，Agent 会失去自主性。比较稳的做法是：

- 目标写清楚
- 边界写清楚
- 让它自己决定微观步骤

不要把每一行操作都替它写死，除非任务风险极高。

## Chapter 14 计划先行：让 Agent 先分析，再执行

很多失败并不是因为模型不够强，而是因为它还没看清问题就被要求立刻动手。

“先 plan，再 act”不是仪式感，而是一种损失控制机制。计划能让你在真正修改前先检查三件事：

- 它理解的问题是不是对的
- 它打算改哪里
- 它的验证路径是否靠谱

### 什么任务必须先出计划

以下任务基本都应该强制先 plan：

- 跨多个模块的改动
- 涉及数据库、权限、配置、部署的修改
- 老项目、陌生项目、文档缺失项目
- 任何“错一次成本很高”的任务

### 什么任务可以边做边看

以下任务可以适当放松：

- 批量修格式
- 小范围测试补充
- 明确可逆的机械改动
- 你已经非常熟悉的局部代码操作

### 好计划长什么样

好计划不是“先分析，再修改，再验证”这种废话，而是要具体到：

- 会看哪些文件
- 为什么先看这些
- 预计改动哪些位置
- 用什么命令验证
- 哪些地方不确定

### 多模型组合中的计划角色

这也是为什么强模型常常适合先做规划，而便宜模型适合执行。规划阶段最需要的是正确拆解问题和找准风险点，执行阶段则更需要耐心和低成本试错。

例如：

- 让强模型先产出计划和验收标准
- 再让便宜模型按计划完成重复性修改
- 最后由强模型做 review 或收尾

这不是玄学，而是把成本放在最值钱的位置上。

## Chapter 15 Context Engineering

如果说 prompt 是“任务规格”，那 Context Engineering 就是“给 Agent 准备工作环境”。

新手最常犯的错误是把上下文等同于“聊天历史”。真正的上下文工程包括：

- 仓库规则文件
- 项目 README
- 架构文档
- 设计文档
- 测试入口
- 构建命令
- 历史约束
- 当前任务的临时信息

### 哪些信息适合长期固化

以下内容适合放进长期规则或记忆文件：

- 项目目录结构和模块边界
- 构建、测试、lint 命令
- 代码风格和命名规范
- 常见坑和禁改区域
- 团队约定的 Git / PR / 发布流程

### 哪些信息应该临时注入

以下内容更适合作为一次性任务上下文：

- 当前需求背景
- 临时 workaround
- 某个 issue 的具体讨论
- 某次异常日志
- 某个临时分支上的实验性约束

### 常见规则文件的角色

- `AGENTS.md`：适合放面向 Agent 的项目工作规则
- `CLAUDE.md`：适合 Claude Code 的长期项目与个人约束
- Cursor Rules：适合按目录、文件类型、模式做精细化规则
- Copilot Instructions：适合平台或仓库级别的辅助约束

这些文件的共同原则是：简洁、明确、无冲突。规则写得像公司制度大全，Agent 通常执行得更差。

### Context 不要越堆越厚

很多团队一开始很兴奋，恨不得把所有规范都塞进规则文件。几周之后，Agent 开始频繁忽略重点、执行混乱。原因很简单：上下文本身也是资源。你要的是高信噪比，不是资料堆积。

## Chapter 15.5 上下文工程七大模式（TODO：后续补图与短例子）

在 xhs notes 和 Anthropic 的上下文工程文章里，有一组非常适合直接做成教学骨架的模式。这里先给出简要说明，后续可以在正式稿中为每个模式配一张图和一个迷你示例。

### 模式一：给 Agent 一台计算机

关键想法是：

- 不要一上来就设计几十个 RPC 工具，而是像给人类开发者那样，先给 Agent 一台“能访问文件系统、能跑命令、能发 HTTP 请求”的计算机；
- 让它在这个标准环境里读写文件、执行脚本、查看日志，而不是把所有事情都封装成抽象工具。

这就是为什么本书以 Claude Code 一类“终端型 Agent”作为主线：它们把最基础的行动空间暴露得够清晰，也最适合教学。

### 模式二：分层的行动空间

和前面 Part 2 提到的三层结构呼应：

- 第 1 层：本地文件系统 & CLI；  
- 第 2 层：协议型工具（MCP / HTTP / SDK）；  
- 第 3 层：多 Agent / 云端服务。

设计建议是：优先把第 1 层打磨好，再逐步往外扩。不要在本地环境都没跑顺的情况下，就急着上多 Agent 和复杂协议。

### 模式三：上下文卸载（Offloading）

比喻是：人类做复杂项目时，会把中间结果写在纸上，而不是全记在脑子里。

对 Agent 而言：

- 不要把所有中间对话、长日志、临时结论都塞在同一次上下文里；
- 更好的做法是把阶段性结果写入文件（如 `logs/`、`notes/`、`REPORT.md`），在需要时再按需载入关键部分。

这样既能保留可追溯性，又不会让上下文越来越臃肿。

### 模式四：上下文缓存（Caching）

生产环境里，极大一部分成本来自“重复加载同样的内容”，包括：

- 系统提示；
- 工具定义；
- 项目规则与说明；
- 常用脚本和命令。

合理的上下文缓存策略可以显著降低成本，同时让 Agent 的“前缀”更稳定。实践上通常包括：

- 把相对稳定的部分放在单独文件里，需要时引用或裁剪；
- 在调用层面尽量保持前缀稳定，只在后缀追加任务相关内容；
- 对多轮任务使用同一套缓存，而不是每轮都重构前缀。

### 模式五：上下文隔离与多 Agent

当一个 Agent 的动作范围过大、上下文已经难以管理时，很多团队会考虑引入多 Agent。但这里有一个重要前提：

- 多 Agent 是用来隔离职责和上下文的，不是用来“凑热闹”的。

适合用多 Agent 的场景包括：

- Planner / Executor / Reviewer 各自有清晰职责；
- 不同子系统之间几乎没有共享状态，只需要汇总结果；
- 可以用 Map-Reduce 思路拆分任务。

不适合用多 Agent 的场景则包括：

- 子任务间高度耦合、频繁共享细粒度状态；
- 上下文本身就已经难以管理，再拆只会增加同步开销。

### 模式六：上下文演化（Context Evolution）

更深一层的问题是：Agent 能否从既往任务中“学到东西”，让后续任务越来越轻松？

一个相对健康的演化路径是：

1. 任务结束后，由 Agent 或人类写一份简短的经验总结；  
2. 对这份总结进行人工或自动审核，去掉错误经验和偶然性结论；  
3. 把可靠内容写入 `CLAUDE.md`/`AGENTS.md`/规则文档；  
4. 对重复性强的流程，进一步提炼为 Skill 或脚本；  
5. 定期清理这些记忆文件，避免旧经验变成负担。

这样做的结果是：Agent 的“长期记忆”在有监督地演化，而不是无序堆积。

## Chapter 16 Verification Engineering

很多教程写到这一步会突然失去工程味，只剩一句“最后记得检查一下”。这远远不够。对 Agent 来说，验证不是补充动作，而是任务闭环的一部分。

### 验证为什么比“看起来对”更重要

模型特别擅长生成“像对的东西”。如果没有真实环境反馈，它会高估自己的正确率。真正可靠的 Agent 工作流必须把验证嵌进任务说明里。

### 可用的验证手段

- 单元测试
- 集成测试
- build
- lint
- 类型检查
- 截图对比
- 浏览器控制台检查
- 接口返回与日志比对

### 好的完成条件怎么写

不要写：

“改完后确认没问题。”

要写：

“改完后运行 `npm test -- login` 或等价验证命令，若失败则继续修复直到通过。最后说明改了哪些文件、执行了什么验证、还有哪些未覆盖风险。”

### 失败后怎么继续

当验证失败时，不要只把一整个长日志重新丢回去。更好的方式是：

1. 保留关键错误片段。
2. 要求 Agent 基于这个错误重新定位。
3. 要求它说明前一次判断哪里错了。
4. 再进入下一轮修改。

### 最值得固定下来的一个习惯

每个任务的 prompt 里都显式写出：

- 验证命令
- 完成标准
- 失败后的处理方式

这会显著减少“看起来做完了，实际上没跑过”的假完成。

## Agentic Coding 的工程原则（理论小结）

结合业界对 Agentic Workflows / Agentic Coding 的讨论，可以把这一部分的方法论浓缩成几条工程原则，作为你设计自己工作流时的检查清单：

- 开发者负责制：无论 Agent 多强，最终对变更结果负责的一定是人类开发者。不要把责任外包给模型，把它当成协作工具，而不是决策主体。
- 理解先于修改：在动手改代码之前，Agent 应该先证明自己理解了问题、上下文和约束。规划阶段的质量，决定后面执行阶段的上限。
- 验证内嵌流程：测试、build、lint、截图、日志检查不是“可选附加”，而是工作流的一部分。没有验证的“完成”，都应该被视为未完成。
- 人类在环：关键节点要保留人工判断——例如计划确认、高风险改动审批、生产操作授权等。把人从机械劳动中解放出来，而不是从责任中移除。
- 抽象边界清晰：Agent 不是要替代所有抽象，而是要在清晰的模块边界内工作。让它围绕清楚的 API、模块、服务边界做事，比让它在一团混乱的代码里“自由发挥”稳定得多。
- 信息密度优先：Prompt、Context、Memory 都应该追求高信噪比，而不是堆砌。给 Agent 提供“决定问题的那 20% 信息”，比给它 100% 原始信息常常更有效。

## Chapter 16.5 从 Prompt Engineering 到 Context Engineering 的观念迁移

很多人一开始是从 Prompt Engineering 入门的：学习如何写一段更清晰的指令，让模型少犯错、多输出一点有用内容。这当然有价值，但在 Agentic Coding 语境下，只做 Prompt Engineering 很容易遇到几个天花板：

- 模型理解得再清楚，也不知道该看哪些文件、跑哪些命令；
- 提示词写得再细，也挡不住上下文越来越乱；
- 单次对话再完美，也难以支撑需要多轮迭代的长任务。

Context Engineering 想解决的，就是“把模型放进一个合适的世界里”，而不仅仅是“对它说一句话”。

可以粗略这样对比：

- Prompt Engineering：像老师给学生下达一条条指令，强调“怎么说”；  
- Context Engineering：像产品经理/项目经理设计整个工作场景，强调“准备什么、限制什么、记住什么、如何演化”。

在 Agentic Coding 中，人类最重要的角色其实更像后者：

- 决定 Agent 拥有什么工具和权限；  
- 决定它看到哪些长期规则和短期上下文；  
- 决定每次任务之后哪些经验被保留，哪些被丢弃。

当你把注意力从“把这句 prompt 写到完美”转向“把整个工作空间设计得合理”，往往会发现：  

- Agent 更少走弯路；  
- 长任务更可控；  
- 新成员接手你的 Agent 工作流也更容易上手。

本 Part 的全部内容，其实都在为这次观念迁移服务：让你不再只盯着那一行输入框，而是学会把 Agent 当成一个需要环境、规则、工具和验证的工程协作者。

## 进一步阅读

想把本 Part 的方法论吃得更透，推荐优先阅读这些资料（配合本书使用，而不是替代本书）：

- Anthropic 官方博客：Agentic Coding & Context Engineering 系列  
  - 例如“Building effective agents”“Effective context engineering for AI agents”“Introducing Contextual Retrieval”等文章，用来加深对计划、工具使用和上下文治理的理解。  
- GitHub 文章：[Software development with agentic workflows](https://github.com/resources/articles/ai/software-development-with-agentic-workflows)  
  - 从平台和团队视角看“Agentic workflows”，对照本 Part 的工程原则，思考你所在团队在哪些环节可以先做小步试点。  
- 社区项目：  
  - [SuperClaude Framework](https://github.com/SuperClaude-Org/SuperClaude_Framework)：观察它如何把“先 plan 再 act”“命令体系化”落到具体配置和脚本上。  
  - [claude-mem](https://github.com/thedotmack/claude-mem)：关注它如何管理长期记忆和上下文污染问题，对照本 Part 的 Memory 与 Context Engineering 章节。  

阅读时可以刻意带着两三个问题：  

- 这些文章/项目在做的，属于 Prompt、Context 还是 Verification？  
- 能否抽出一条对你自己的项目立刻有用的规则，写进 `AGENTS.md` 或 `CLAUDE.md`？  
- 哪些做法适合你现在的规模，哪些应该等到“有更稳定的基础工作流后再考虑”？  


如果你在设计任何一个 Agent 工作流时都能对照这几条做自检，这一部分的方法论就已经真正落地，而不是停留在抽象概念上。
