# Part 1 快速上手

这一部分的目标很明确：不要让读者在理解完概念之前就已经失去耐心。Agentic Coding 教程如果不能让读者在前 30 分钟内真正跑通一次任务，那后面再多原理都很难建立信心。

## Chapter 4 新手最短路径

新手最容易犯的错不是“不会用”，而是“同时装太多工具，结果每个都只会一点点”。因此最短路径不是产品越多越好，而是先选一条主线，跑通一个最小闭环。

推荐三条起步路线：

### 路线 A：`Claude Code`

适合这类人：

- 习惯终端工作流
- 想先理解 Agentic Coding 的原始形态
- 更愿意从“任务闭环”而不是“IDE 便利性”切入

最短流程可以写成：

1. 准备账号与环境。
2. 安装 `Claude Code`。
3. 进入一个真实项目目录。
4. 让它先读仓库，再给计划。
5. 让它做一次小修改并验证。

官方文档当前给出的快速安装路径是 `Node.js 18+` 后全局安装，再进入项目目录运行 `claude`。第一次登录后，建议不要一上来就丢一个“大而全”的需求，而是给一个小而完整的任务，比如：

```text
先阅读这个仓库，告诉我项目结构、启动命令、测试命令和最值得优先了解的三个模块。
不要修改代码，先给出你的判断。
```

第二个任务再升级为：

```text
请在理解仓库后，修复一个最小问题：
1. 找出一个低风险、容易验证的小问题
2. 先给计划
3. 修改后运行相应验证命令
4. 最后总结改动和验证结果
```

### 路线 B：`Codex`

适合这类人：

- 更偏好 OpenAI 生态
- 希望同时理解本地 CLI Agent 和云端 Agent 的差别
- 想把“并行任务”和“云沙箱”也纳入后续工作流

当前公开资料显示，`Codex` 同时存在 CLI、本地 IDE 接入和云端 Web Agent 形态。对新手来说，先从 CLI 或桌面侧起步最稳，因为你更容易建立“本地仓库 + 明确目标 + 回报验证”的基础感知。新手第一次使用时，同样建议把任务控制在一个小闭环内，不要上来就让它“接管整个项目”。

第一轮任务可以直接照搬上面的仓库理解模板。第二轮则可以换成：

```text
阅读这个仓库后，帮我补一条最小回归测试。
要求：
1. 不改业务逻辑
2. 先说明你选中的模块和原因
3. 修改后运行测试验证
4. 输出简短总结
```

### 路线 C：`Cursor` 或 `Cline`

适合这类人：

- 更依赖 IDE 可视化交互
- 喜欢边看 diff 边指导 Agent
- 需要更平滑地从“编辑器 AI”过渡到“Agent 工作流”

`Cursor` 的优势是把 Agent 和代码编辑体验整合得很自然，适合从现有编辑器习惯平滑升级。`Cline` 的优势是人机协同边界很直观，很多动作都明确要求你审批，适合把“控制权”和“自动化”之间的关系看清楚。

第一次上手时，推荐你做这三件事：

1. 只给 Agent 一个很窄的任务范围。
2. 明确告诉它先分析，再执行。
3. 必须要求它在完成前运行验证动作。

### 新手第一周的唯一目标

不要试图一周内学完所有工具。第一周真正要达成的不是“我会很多命令”，而是这三件事：

- 你已经体验过 Agent 读仓库、改文件、跑命令、继续修。
- 你知道什么时候该先让它出计划。
- 你开始明白为什么验证和上下文比模型名更重要。

## Chapter 5 部署与接入：官方账号、API、中转商

这是最容易把新人带偏的一章，因为大家最先搜到的往往不是“原理”，而是“谁家便宜、谁家能转、谁家能逆向接”。如果一开始就只盯着价格，很容易在稳定性、权限和安全上吃大亏。

建议把接入方式分成四层来理解：

### 1. 官方支持方案

这是最优先推荐的路径。优点是：

- 兼容性最好
- 问题排查最直接
- 产品特性支持最完整
- 安全、权限和日志边界更清楚

缺点通常是价格不一定最低，区域和套餐限制也可能更多。

### 2. 官方兼容云方案

例如云厂商托管的模型接入层。这类方案的优点是企业治理、网络合规、身份体系更成熟，适合团队场景。缺点是接入配置更重，某些产品能力上线速度可能慢于原厂。

### 3. 兼容 API 聚合层

这类服务通过统一接口暴露多个模型或多个提供商。它的价值在于：

- 可以用一个 `Base URL` 和一组凭证接多家模型
- 便于切换供应商和模型
- 常常有路由、重试、监控、计费聚合等附加能力

它的风险也很明显：

- 模型 ID 映射未必和官方一致
- 上下文长度、工具能力、缓存行为可能与官方不同
- 发生故障时，你很难判断问题是在 Agent、聚合层还是底层 provider
- 日志、隐私、风控和限速策略不一定透明

### 4. 非官方逆向方案

这类方案短期内看起来很香，因为门槛低、价格可能更低、配置看起来也简单。但它的问题不是“道德争议”这么抽象，而是工程上很难稳定依赖：

- 账号或接口随时失效
- 风控规则不透明
- 模型能力可能被改写或阉割
- 出问题时几乎没有正式支持
- 对个人和团队安全都是隐患

### `Base URL` 和 `API Token` 到底是什么

很多新手在配置界面里只会机械地复制粘贴，不理解背后的链路。其实可以把它想得很简单：

- `Base URL` 决定“请求发到哪台服务”
- `API Token` 决定“你是谁、你有没有权限、按谁计费”

图示建议：

`Agent Client -> Provider SDK / Compatible API -> Model Endpoint -> Tool / Result`

理解这一点后，你就会明白为什么同一个 Agent 前端可以切不同后端，也会明白为什么“能填进去不代表真的兼容”。真正兼容的，不只是文本接口，还有：

- 上下文窗口行为
- 工具调用风格
- 速率限制
- 输出长度
- 权限模型
- 日志与审计链路

### 给新手的部署建议

如果你是个人开发者，最稳的顺序是：

1. 先走官方支持路径。
2. 熟悉工作流后，再尝试兼容 API。
3. 真要用聚合层，也优先选文档完整、模型映射透明、日志说明清楚的方案。
4. 不要把逆向方案当成长期主力。

如果你是团队或公司，更稳的顺序是：

1. 先明确数据分级、权限边界和审计要求。
2. 再决定走原厂、云厂商托管，还是统一网关。
3. 最后才是讨论价格和费率组。

## Chapter 6 第一个真正有价值的任务

很多教程的问题是第一个例子太假：要么是从零生成一个 Hello World，要么是写一个根本不会进入真实工作流的小 demo。更好的做法是，第一任务就选真实仓库里的真实小问题。

推荐的标准任务是：

“接手一个现有仓库，理解结构，完成一个低风险、小范围、可验证的修改。”

这个任务之所以好，是因为它天然包含了 Agentic Coding 的完整闭环：

1. 读取代码库
2. 理解结构
3. 给出计划
4. 修改代码
5. 执行验证
6. 总结结果

你可以直接把第一次任务写成下面这个模板：

```text
你现在在一个已有项目中工作。

目标：
先理解仓库，再完成一个最小但真实的改动。

要求：
1. 先浏览项目结构，告诉我入口、主要模块、启动命令、测试命令
2. 基于理解，选择一个低风险的小改动
3. 先给出计划，不要立刻改
4. 我确认后再执行修改
5. 修改完成后运行相应验证命令
6. 输出改动摘要、涉及文件、验证结果、剩余风险
```

如果你不想分两轮确认，也可以直接要求：

```text
先分析，再执行，但执行前要先把计划写出来。遇到不确定的地方停止并说明。
```

### 为什么这个任务比“写一个新页面”更好

因为新页面生成更像是拼装题，而理解现有仓库才是现实开发中最消耗认知成本的部分。Agent 的真正价值，很多时候不在从零生成，而在：

- 接手陌生代码
- 跨文件追踪逻辑
- 处理真实命令输出
- 根据验证结果继续迭代

### 第一次任务的验收标准

第一次就要建立“完成”的硬标准，不要让 Agent 只给你一个看起来像样的 diff。

至少要检查：

- 它是否真的读过关键文件，而不是靠猜。
- 它的计划是否具体到文件和动作。
- 它是否真的执行了验证命令。
- 它是否解释了没验证到的风险。

### 本章最重要的习惯

你以后几乎所有任务都应该默认带上三句话：

- 先分析再执行。
- 修改后必须验证。
- 如果不确定，就停下来说明。

这三句看起来朴素，但它们会直接决定你后续 80% 的 Agent 使用质量。

## 进一步阅读与练习建议

如果你希望在“第一次跑通”之后继续加深理解，可以结合本章内容阅读和练习以下资料：

- Anthropic 官方文章：[Introducing agentic coding](https://www.anthropic.com/engineering/claude-code-best-practices)  
  - 重点关注它如何定义 Agentic Coding，以及对“目标 → 计划 → 执行 → 验证”闭环的拆解。  
- 文档：[Claude Code overview](https://docs.anthropic.com/en/docs/claude-code/overview) 与 [common workflows](https://docs.anthropic.com/en/docs/claude-code/common-workflows)  
  - 对照本章的最短任务，把官方给出的“读仓库 / 修改 / 验证”示例看一遍，找到两者的共同模式。  
- Cursor Docs: [Agent](https://docs.cursor.com/agent) 与 [Rules](https://docs.cursor.com/context/rules)  
  - 尝试用本章的“第一个任务模板”在 Cursor 里跑一遍，并把规则写进 Cursor Rules，而不是只写在 prompt。  
- Cline Docs: [Cline 使用文档](https://docs.cline.bot/)  
  - 重点体会它的人机审批流，在第二个或第三个任务中刻意练习“先计划、再审批、再执行”的节奏。  

练习建议：

1. 选一个你熟悉的小项目，照着本章模板从头到尾跑完一次。  
2. 再选一个你不太熟悉的项目，重复同样流程，并记录“陌生项目”和“熟悉项目”的 Agent 体验差异。  
3. 把你认为最有用的两条 prompt/命令写进项目的 `AGENTS.md` 或 `CLAUDE.md`，作为未来每次接手新任务时的起步模板。
