# Part 6 MCP、Skills 与插件生态

到这里，读者应该已经不满足于“直接对一个 Agent 下命令”了。真正的进阶阶段一定会遇到两个问题：第一，怎么给 Agent 连接外部能力；第二，怎么把自己摸索出来的做法沉淀成可复用资产。MCP 和 Skills 正好分别对应这两个方向。

## Chapter 21 MCP 到底解决了什么问题

MCP 最容易被误解。有人把它当成新模型，有人把它当成某家的专属插件标准，还有人觉得它只是“给 AI 接数据库的协议”。这些理解都太窄了。

更准确地说，MCP 是一层标准化的能力连接协议。它解决的是：模型或 Agent 想安全、结构化、可发现地访问外部工具和数据源时，双方如何说同一种话。

一定要先讲清三件事：

- MCP 不是模型
- MCP 不是 Agent 本身
- MCP 是 Agent 与外部能力之间的协议层

### 为什么它重要

如果没有类似 MCP 的标准，每接一个外部能力都要做一套私有集成：

- 不同工具不同接口
- 不同权限模型
- 不同返回格式
- 不同发现方式

这会让 Agent 生态碎片化得非常厉害。MCP 的价值就在于把“接能力”这件事尽量标准化。

### 本地 MCP 与远程 MCP

本地 MCP 更适合：

- 本地文件系统
- 本地数据库
- 本地浏览器
- 对数据出域敏感的场景

远程 MCP 更适合：

- 团队共享服务
- 知识库
- GitHub、工单系统、设计平台等云服务
- 需要集中治理和统一鉴权的场景

### 常见 MCP 场景

- 浏览器自动化
- GitHub 仓库与 issue 操作
- 数据库查询
- 文档与知识库检索
- 设计稿和原型读取
- 内部平台和工单系统接入

教程里一定要明确：MCP 不是为了炫酷，而是为了把 Agent 从“只能读本地代码”扩展到“能在受控边界内接入真实工作环境”。

## Chapter 22 Skills、MCP、插件、脚本的关系

这一章必须把概念分开，否则读者很快会把所有东西都叫插件。

可以直接给一张对比表：

| 形态 | 核心价值 | 更像什么 | 适合场景 |
| --- | --- | --- | --- |
| Skill | 复用经验、流程和提示结构 | 方法论模板 | 固定工作流、固定套路 |
| MCP | 暴露外部能力和数据源 | 标准化接口 | 浏览器、数据库、GitHub、知识库 |
| 插件 | 产品层集成和体验封装 | 应用扩展 | IDE、客户端、平台集成 |
| 脚本 | 确定性自动化动作 | 命令工具 | 格式化、测试、构建、批处理 |

### Skill 的本质是“教方法”

一个 Skill 最有价值的时候，不是它写了很多字，而是它能把：

- 触发条件
- 操作步骤
- 使用模板
- 输出要求

打包成一个可重复执行的单元。

### MCP 的本质是“给能力”

MCP 不关心你写 PR 描述的语气是否专业，它关心的是 Agent 能不能安全地访问 GitHub、浏览器、数据库或知识库。

### 为什么脚本没有过时

很多人一学 MCP 和 Skill，就觉得脚本不重要了。恰恰相反。脚本仍然是最可靠的确定性执行单元。越成熟的 Agent 工作流，越会把“开放式推理”与“确定性脚本”结合起来，而不是让模型自由发挥到底。

## Chapter 23 Skill 开发入门

写 Skill 时，最常见的问题不是“不够强”，而是“不够聚焦”。很多 Skill 试图覆盖十几种场景，结果哪种都不够稳定。

一个好 Skill 至少要回答四个问题：

- 它什么时候被触发
- 它帮 Agent 做什么
- 它需要哪些输入
- 它最终交付什么

### 一个推荐的 Skill 目录结构

```text
my-skill/
  SKILL.md
  scripts/
  templates/
  references/
```

其中：

- `SKILL.md` 负责定义触发条件、工作流、注意事项
- `scripts/` 放确定性脚本
- `templates/` 放可复用模板
- `references/` 放必要的补充材料

### `SKILL.md` 应该写什么

最重要的是让 Agent 快速进入正确工作方式，而不是写成长篇小说。通常包括：

- 这个 Skill 解决什么问题
- 什么时候该用
- 什么时候不该用
- 推荐步骤
- 需要查看哪些文件
- 需要调用哪些脚本或模板
- 输出格式要求

### 如何减少无关上下文

Skill 的意义之一就是节省上下文。如果一个 Skill 自己就带来大量噪音，那它会适得其反。做法包括：

- 只写对该类任务稳定成立的规则
- 把长篇资料放进 `references/`，按需引用
- 把可执行逻辑交给 `scripts/`
- 把格式要求交给 `templates/`

### 用 `AgentSkills` 作为案例

你自己的 `AgentSkills` 非常适合拿来写“meta skill 引导开发新 skill”。这是一个很好的切入点，因为它天然体现了 Skill 体系的递归价值：

- 先用一个 Skill 帮你澄清目标
- 再帮你拆结构
- 再帮你产出适合长期复用的 Skill 说明

这一章未来可以直接加一个小案例：从“我想给前端性能诊断做一个 Skill”开始，到最终产出可用的 `SKILL.md`。

## Chapter 24 MCP 开发入门

讲 MCP 开发时，建议不要一上来就让读者自己写 server。更好的顺序是：

1. 先安装并使用一个现成 MCP
2. 理解它如何暴露能力
3. 再去写自己的最小 MCP server

### 先从安装现成 MCP 开始

这一步的意义是让读者先知道“一个可用的 MCP 到底长什么样”。比如：

- 它暴露哪些工具
- 需要什么配置
- 如何鉴权
- 调用结果以什么格式返回

### 自己写 MCP 时要先想清楚三件事

- 你要暴露的是工具、资源，还是工作流入口
- 权限边界在哪里
- 哪些动作必须是只读，哪些动作允许写入

### 最小 MCP Server 的设计原则

- 能力少而清晰
- 返回结构稳定
- 错误信息可读
- 权限模型明确
- 审计和日志有基本设计

### 什么时候适合做成 MCP

以下场景很适合：

- 你的团队有一个很多 Agent 都想调用的能力
- 它背后连接的是外部系统
- 它需要统一权限和鉴权
- 它值得长期维护

以下场景则不一定需要 MCP：

- 只是你个人偶尔用一次的小脚本
- 完全本地、无需共享的单次自动化
- 规则和提示就能解决的问题

Skill、脚本、MCP 三者没有高低之分，只有是否匹配问题。

## Chapter 24.5 什么时候只用 CLI，就够了（TODO：补一两个判断示例）

在 xhs notes 的视角下，有一个很重要但常被忽略的判断：并不是所有外部能力都要做成 MCP 或插件，很多时候一个简单的 CLI 命令就足够了。

可以先问自己三个问题：

1. 这个能力是否已经有成熟的命令行工具或脚本？  
2. 它是否只在本地使用、不需要多人共享？  
3. 它的风险边界是否可以通过终端权限和工作目录很好控制？

如果以上问题大多回答“是”，那么优先用 CLI 往往更划算：

- 实现成本低：写一个脚本远比写一个 MCP server 简单；  
- 调试更直观：你可以直接在终端复现同一命令；  
- 复用度高：CI、开发者本地、Agent 都可以调用同一个脚本。

只有在以下场景时，才更应该考虑 MCP：

- 需要在多台机器、多种 Agent 上共享同一套能力；  
- 需要统一鉴权、配额和审计；  
- 能力背后连接的是外部系统（GitHub、工单系统、内部平台等）。

换句话说：

- 本地一次性的“小工具” → 脚本 / CLI；  
- 团队共享的“基础设施能力” → MCP；  
- 跨产品的“交互体验封装” → 插件；  
- 可复用的“做事方式” → Skill。
