# Part 8 实战案例

这一部分的任务不是再讲一遍原理，而是把前面的方法落到足够具体的场景里。每个案例都尽量保持统一结构：场景、初始 prompt、计划、关键执行、验证、复盘。等后续补充动图和真实会话记录后，这一部分会变得非常有说服力。

## Case 1 接手陌生仓库并完成首次功能修改

### 场景

你刚 clone 下来一个项目，仓库不小，文档也不算完整。你的目标不是立刻做大改动，而是在最短时间内建立结构感，并完成一个小的真实修改。

### 初始 prompt

```text
先阅读这个仓库，告诉我：
1. 项目的主要模块和入口
2. 启动、测试、构建命令
3. 哪个模块最适合做第一次低风险修改

先不要修改代码，先给出计划。
```

### 理想计划

- 浏览根目录和 README
- 查 package / build / test 入口
- 搜索最基础的页面或接口模块
- 选择一个低风险点作为首次修改对象

### 关键执行

第一次改动可以是：

- 修复一个明显文案错误
- 补一个缺失默认值
- 修一个简单边界 bug
- 为一个已有函数补单测

### 验证

- 运行对应测试或构建命令
- 说明验证结果
- 说明未覆盖风险

### 复盘

这个案例的重点不是改了什么，而是让读者第一次感受到 Agent 真正在“接手项目”，而不是只生成代码片段。

## Case 2 修一个跨前后端的复杂 bug

### 场景

用户反馈“订单页有时会显示成功，但后台实际上没有写入记录”。这类问题天然适合展示 Agent 的优势，因为它需要跨前后端追踪逻辑、查看日志、阅读多个模块。

### 初始 prompt

```text
请帮我定位“订单页偶发显示成功但后台未持久化”的原因。

要求：
1. 先分析调用链和相关模块
2. 给出最可能的 2 到 3 个原因
3. 说明你打算先验证哪一个，为什么
4. 在我确认后再修改
```

### 理想计划

- 找前端提交逻辑
- 找对应接口
- 找服务端持久化链路
- 检查异常处理与返回值映射
- 确定是否存在“前端把局部成功当整体成功”的逻辑漏洞

### 关键执行

这个案例里，Agent 的真正价值往往是：

- 能跨目录快速建立调用链
- 能把多个猜想排优先级
- 能通过日志或测试逐步排除错误路径

### 验证

- 增加或运行一条回归测试
- 模拟失败分支
- 确认前端提示与后端状态一致

### 复盘

这是非常适合展示“先计划再执行”和“验证工程”的案例。读者会看到，复杂 bug 修复不是靠神 prompt，而是靠证据链。

## Case 3 为遗留项目补测试并顺手清理技术债

### 场景

遗留项目最怕两件事：没人敢动，和每次一动就出新问题。Agent 在这里最适合做的不是大刀阔斧重构，而是先把保护网补起来。

### 初始 prompt

```text
请先识别这个仓库中最缺测试保护的一个模块。

要求：
1. 不做大重构
2. 先说明为什么选这个模块
3. 先补一条最有价值的回归测试
4. 如果能在不扩大风险的前提下顺手清理一处小技术债，也请说明
```

### 理想计划

- 找无测试或薄弱测试的关键模块
- 选一个容易验证且业务价值高的分支
- 先补测试，再做小修复

### 关键执行

这个案例特别适合体现一种成熟姿势：先让测试成为安全边界，再谈重构。

### 验证

- 新增测试通过
- 旧测试不回归
- 变更范围控制在最小

### 复盘

很多团队最需要的并不是“AI 帮我写新功能”，而是“AI 帮我一点点把遗留系统拉回可维护状态”。

## Case 4 从 PRD 或 issue 出发实现一个小功能

### 场景

这类案例最贴近日常研发：你有一个产品需求描述或一个 issue，需要把它从自然语言转成代码和验证。

### 初始 prompt

```text
下面是一条需求，请先把它转成工程任务。

要求：
1. 先总结需求
2. 给出涉及模块
3. 给出最小实现计划
4. 标出不明确点
5. 我确认后再开始改
```

### 理想计划

- 把需求从自然语言翻译成界面、接口、状态、校验、测试几个维度
- 明确边界条件
- 明确哪些是当前迭代不做的内容

### 关键执行

这一类案例里，最适合强调的是“需求澄清”。Agent 不是需求产品经理的替代品，但它很适合把模糊描述翻译成工程任务草案。

### 验证

- 对应测试
- 手工验证路径
- 变更说明与已知限制

### 复盘

如果这个案例写得好，读者会理解为什么 Agent 在“从 issue 到实现”这条链路上价值极大。

## Case 5 安装并使用一个 MCP，然后开发一个简单 Skill

### 场景

这是一条最能体现“生态扩展”的案例。目标是让读者看到，Agentic Coding 不只是和代码打交道，还能逐步连接外部能力和复用流程。

### 初始 prompt

```text
请先帮我安装并验证一个现成 MCP，再基于这个项目需求设计一个最小 Skill。

要求：
1. 先说明 MCP 解决什么问题
2. 安装后做一次最小调用验证
3. 再设计一个对应的 Skill
4. Skill 只覆盖一个单一任务
```

### 理想计划

- 选择一个简单的 MCP，如浏览器或 GitHub 相关能力
- 完成最小配置
- 用一次真实调用验证它能工作
- 基于项目需求，把常见流程写成 Skill

### 关键执行

这个案例里最值得展示的是边界意识：

- MCP 提供能力
- Skill 提供方法
- 脚本提供确定性动作

### 验证

- MCP 工具调用成功
- Skill 能在正确场景下被触发并产出预期输出

### 复盘

如果前几章是教读者“如何使用别人的 Agent”，这个案例开始教读者“如何扩展自己的 Agent 系统”。

## 每个案例结尾都要有“经验沉淀”小节

前面的 5 个案例，更多是在展示“怎么用 Agent 把事情做成”。要把这部分真正升级成 Agentic Coding 教程的一部分，还需要在每个案例里固定加入一个短小但关键的环节：经验沉淀。

建议为每个案例增加类似的收尾小节，名字可以统一叫做：

- 「经验沉淀与系统演化」

内容可以包括三类问题：

1. 这次任务对 `AGENTS.md` / `CLAUDE.md` / 规则文件有什么补充？  
2. 有哪些步骤值得被提炼成 Skill 或脚本？  
3. 有哪些错误经验需要刻意避免写进长期记忆？

这样做的直接收益是：

- 让读者看到“任务结束后该做什么”，而不是停在“完成一次就结束”；  
- 把“Context Evolution / 上下文演化”的思想嵌入最实际的案例里，而不是只停留在 Part 4 的理论段落；  
- 为团队级落地留下一条清晰路径：每次任务之后，系统都会比之前稍微好一点。

> TODO：后续在具体扩写每个案例时，可以先以段落形式在最后加上「经验沉淀与系统演化」小节，再视情况演化成对应的 `AGENTS.md` 片段、Skill 示例或脚本片段。

## 进一步阅读与案例取材

在补充和细化本 Part 的每个案例时，可以从这些资料中抽取灵感和素材：

- Anthropic / Claude Code  
  - [Claude Code common workflows](https://docs.anthropic.com/en/docs/claude-code/common-workflows)：从中挑选与你的案例类似的场景（如“修 bug”“补测试”“文档整理”），对照官方流程设计自己的案例步骤。  
  - [Introducing agentic coding](https://www.anthropic.com/engineering/claude-code-best-practices)：用作“案例复盘”的理论背景，尤其是对任务闭环和验证的强调。  

- Agentic workflows 与团队实践  
  - GitHub 文章：[Software development with agentic workflows](https://github.com/resources/articles/ai/software-development-with-agentic-workflows)：从团队视角看“从 issue 到 PR”的链路，为 Case 4“从 PRD/issue 出发实现功能”提供结构参考。  

- 社区实践与教程  
  - [SuperClaude Framework](https://github.com/SuperClaude-Org/SuperClaude_Framework)：观察它如何记录真实任务过程与命令，借鉴到你自己的“执行日志 + 复盘”写法上。  
  - 你自己的 [AgentSkills](https://github.com/zht043/AgentSkills)：从中挑选 1–2 个 Skill 场景，改编成“先用 Skill 设计任务，再用 Agent 执行”的案例。  

建议在写每个案例时，都从真实项目/真实任务中截取素材，而不是凭空设计 demo，这样读者在照搬时更容易少踩坑。
