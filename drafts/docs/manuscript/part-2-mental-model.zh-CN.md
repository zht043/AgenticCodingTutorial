# Part 2 Agent 心智模型

如果 Part 1 解决的是“先跑通一次”，那这一部分解决的就是“为什么有时候很好用，有时候又突然变笨”。很多人把 Agent 当作黑箱来用，结果一旦输出质量下降，就只能归因于模型不行。实际上，很多问题都来自心智模型不清。

## Chapter 7 Chatbot、代码补全、IDE 助手、Code Agent 的区别

不是所有带 AI 的工具都应该被叫作 Agent。这个区分如果不做清楚，后面的对比就会变成鸡同鸭讲。

可以先看一张简化矩阵：

| 类型 | 典型代表 | 能力中心 | 是否能操作环境 | 是否适合长任务 |
| --- | --- | --- | --- | --- |
| Chatbot | 通用聊天产品 | 问答、解释、生成 | 通常不能 | 一般 |
| 代码补全 | IDE 自动补全 | 局部代码预测 | 不能 | 不适合 |
| IDE 助手 | IDE Chat / Inline Edit | 面向当前文件或局部上下文辅助 | 部分可以 | 中等 |
| Code Agent | Claude Code、Codex、Cline、Cursor Agent | 面向任务闭环的持续执行 | 可以 | 适合 |
| Cloud Agent | 云端 coding agent | 隔离环境并行执行 | 可以 | 很适合 |

Chatbot 最像一个知识顾问。它可以解释 bug、生成脚本、给你思路，但不会自己去执行。代码补全更像一个高级输入法，帮你加速局部编写。IDE 助手介于中间，已经开始理解仓库上下文，但仍然常常依赖你来切换文件、执行命令、做最后验证。

Code Agent 的本质变化在于，它开始承担“执行器”的角色。你给它的是目标和边界，不只是一个问题。它需要自己把任务拆解成一系列行动，再从环境里拿反馈回来。

这也是为什么很多“AI 很强”的体验，其实只是从 Chatbot 或 IDE 助手的角度看很强，但并不等于它适合承担完整开发任务。

### 一个实用判断题

如果一个工具在面对“修复这个仓库里某个 bug，并给我运行结果”时，主要反应仍然是输出建议和代码片段，那么它更接近助手。如果它能在约束内自己完成查找、修改、运行、回报，那它更接近 Agent。

## Chapter 8 Agent 内部到底发生了什么

从外面看，Agent 好像只是“会自己做事的模型”。但从工程上看，它至少包含六层结构，而且每一层都不是模型自己“凭感觉长出来的”，而是开发者设计出来的。

在展开六层之前，可以先用一句话记住一个闭环：

> Agent = LLM（大脑） + Tools（手脚） + Instructions / Memory（性格与经验） + Verification（成绩单）

大多数 Agent 工作时，都在不断重复同一个循环：

`Goal -> Plan -> Act -> Observe -> Verify -> Iterate`

接下来这六层，就是围绕这个循环展开的。

### 1. Instructions 层

这层包括 system prompt、开发者指令、仓库规则文件、项目约束。它决定 Agent 从一开始就站在什么边界里做事。比如：

- 先 plan 再 act；
- 不要修改未授权目录；
- 所有修改必须运行测试；
- 输出必须包含风险说明；
- 禁止动生产配置和密钥。

这层写得乱，后面所有层都会出问题。很多“Agent 跑偏”其实是 Instructions 写得含糊，让它既像助手又像 Agent，既要小心又要全自动，导致行为内在冲突。

### 2. Context Assembly 层

Agent 不可能把整个仓库永远完整放进上下文，所以它必须动态组装上下文：哪些文件先读、哪些日志要保留、哪些信息要压缩、哪些要丢弃。这是 Agent 成败的核心之一。

这层常见的设计点包括：

- 初次任务时，如何选择要读的入口文件和目录；
- 错误日志是全贴还是只贴末尾几十行；
- 长对话怎么做阶段性总结，而不是全量复读；
- 如何区分长期规则（AGENTS.md / CLAUDE.md）和一次性任务背景。

### 3. Planning 层

面对复杂任务，Agent 往往要先形成一个行动计划：先读什么、先验证什么、哪里有高风险、哪些步骤必须人工确��。没有这层，Agent 很容易一上来就乱改。

典型步骤包括：

- 把目标拆成若干子任务；
- 标出哪些子任务可并行、哪些必须串行；
- 为每个子任务约定“完成定义”（Definition of Done）；
- 在计划里提前写好需要跑哪些命令、看哪些文件。

### 4. Tool Use 层

这是很多人最直观看到的部分：读文件、改文件、跑命令、调用浏览器、访问 Git、访问外部数据源。工具多不等于能力强，关键在于工具调用是否被合理编排。

从 xhs notes 的视角看，最重要的不是“工具协议有多花哨”，而是先把最基础的行动空间打牢：

- 文件系统：读取、创建、修改项目中的文件与目录；
- Shell / CLI：运行测试、构建、脚本、脚手架命令；
- 网络 / HTTP：访问必要的外部 API 或服务。

这三类“标准化工具”决定了 Agent 是否真的像一个工程师，而不仅仅是一个聊天机器人。

### 5. Memory / Summarization 层

当任务变长，原始上下文会迅速膨胀。Agent 需要把已完成阶段压缩成可继续使用的摘要，否则上下文会越来越贵、越来越乱、越来越难遵循。

常见做法包括：

- 每个阶段结束后写一段简短总结，替换掉中间的细节对话；
- 把阶段成果写入独立文件（例如 `notes/`、`logs/`、`REPORT.md`），必要时再按需读取；
- 对长期有用的信息写入 `CLAUDE.md`、`AGENTS.md` 或规则文档，而不是永远留在对话里。

这层既是“记忆”，也是“垃圾回收”：帮 Agent 保留真正有长期价值的东西，把一次性噪音从上下文里清理出去。

### 6. Verification / Recovery 层

这层决定 Agent 是否只是“看起来在工作”，还是“真的在交付可用结果”。验证包括测试、build、lint、截图、日志检查。Recovery 则包括失败后如何回退、如何重新规划、如何不在同一个错误上无限打转。

一个成熟的 Agent 工作流，至少应该回答清楚：

- 本次任务的成功条件是什么？  
- 应该跑哪一组自动化检查？  
- 如果检查失败，Agent 有权限做哪些修复尝试？  
- 哪些失败需要人类介入决策？

### 为什么上下文越多不一定越好

新手特别喜欢一股脑把所有文件、所有日志、所有历史对话塞给 Agent，觉得“信息越全越安全”。现实往往相反。上下文太多会带来三个问题：

- 重要信息被淹没
- 矛盾指令变多
- 模型开始对低价值信息过拟合

更准确的做法不是“尽可能多给”，而是“只给完成当前目标最相关的高密度信息”。

### 为什么任务拆解比提示词辞藻更重要

很多人花大量时间打磨一句华丽 prompt，却不愿意把任务拆成“先定位、再修改、再验证、再总结”这种清晰结构。对 Agent 来说，任务结构远比语气修饰重要。你给它一个清晰闭环，通常比给它十条空泛原则更有用。

## Chapter 8.5 Agent 的分层行动空间（TODO：补示意图）

从“所有 Agent 都是 Coding Agent”的视角看，Agent 的行动空间可以分成三层，从近到远分别是：

1. 本地文件系统与 CLI  
2. 协议型工具（MCP / HTTP API / SDK 等）  
3. 多 Agent / 外部服务协作

### 第一层：本地文件系统与 CLI

这是 Code Agent 的“主战场”，也是 Claude Code 一类工具最擅长的区域。典型动作包括：

- 读取和编辑代码、配置、文档等文件；
- 运行测试、构建、脚本、脚手架命令；
- 在本地项目中创建笔记、日志、报告文件。

如果这层做不好，再复杂的 MCP 或远程工具也很难弥补，因为 Agent 连“读懂和修改当前项目”这件事都没打稳。

### 第二层：协议型工具（MCP / HTTP API / SDK）

当本地能力不够用时，Agent 会通过各种协议访问外部世界，例如：

- 通过 MCP 访问浏览器、数据库、GitHub、知识库；
- 通过 HTTP API 调用内部服务；
- 通过 SDK 调用云端平台。

这层的关键不是数量，而是：

- 能力边界是否清晰（工具到底能做什么、不能做什么）；
- 错误语义是否明确（失败时能否给出可恢复的反馈）；
- 权限和审计是否可控。

### 第三层：多 Agent / 外部服务协作

在更复杂的场景里，单个 Agent 的上下文和责任范围会被拆分成多个子 Agent 或外部服务，例如：

- Planner / Executor / Reviewer 分工合作；
- 多个 Agent 并行处理不同子系统或模块；
- 由云端 Agent 在隔离环境里执行高风险任务，最后以 PR 形式回到主仓库。

这一层的收益很大，但代价也更高：需要额外的调度、状态共享设计和成本控制。因此，一般建议在第一层和第二层已经跑得比较顺的时候，再考虑引入多 Agent。

## Chapter 9 Agent 的边界与失败模式

学 Agentic Coding，真正的分水岭不是第一次成功，而是第一次失败之后你能不能看懂它为什么失败。

最常见的失败模式有七类。

### 1. 上下文污染

表现是：Agent 一直抓着旧结论不放，或者被一大堆无关日志和历史对话带偏。

常见原因：

- 你贴了太多过时信息
- 指令冲突
- 仓库规则文件太长且自相矛盾

应对方法：

- 重开一个干净会话
- 只保留当前任务的必要背景
- 把长期规则和临时任务分离

### 2. Memory 污染

表现是：Agent 学到了错误偏好，并在后续任务里不断重复。

这往往来自：

- 你随手让它“记住”一些只适用于单次任务的策略
- 自动记忆缺乏清理
- 规则文件把临时 workaround 写成长期标准

### 3. 长任务漂移

任务一长，Agent 会忘记原目标，开始不断优化细枝末节，或者转去修与当前任务关系不大的问题。

最有效的对策是分阶段回报。不要把一个两小时任务只用一句话描述完后就放任它跑到底。更好的做法是：

- 先出阶段计划
- 每个阶段结束时做总结
- 明确下一阶段的完成条件

### 4. 并行任务相互干扰

多个 Agent 同时读写同一片代码区域，很容易产生冲突。并行不是越多越好，必须先划清职责和边界。

### 5. stdout 过长吞掉 token

很多新手第一次让 Agent 跑命令时，会把完整构建日志、完整测试输出、完整编译堆栈全量塞回对话。结果成本暴涨，质量下降。

更好的做法是只保留：

- 错误摘要
- 关键堆栈
- 失败模块
- 最后几十行相关输出

### 6. 本地环境与模型假设不一致

模型以为你用的是某个工具链版本、某个目录结构、某个包管理器，但你的真实环境根本不是那样。这个问题在跨语言、老项目、Monorepo 场景尤其常见。

### 7. 权限不足或权限过大

权限太小，Agent 干不了活；权限太大，它可能执行超出你预期的操作。真正成熟的用法不是一味放权，而是根据任务风险动态调节。

本章的核心结论是：Agent 不是稳定的自动驾驶，它更像一个很强但需要工程护栏的协作者。你越早接受这一点，越不容易在遇到失败时用“模型又抽风了”草率结束分析。

## 小结：Agentic Coding vs Vibe Coding

这几年社区里还出现了一个词叫 “Vibe Coding”，大致指的是：开发者并不真正理解代码意图，而是把大量生成的代码当作黑箱堆砌，只要“看起来能跑”就接受。这种用法短期内可能让人感觉很爽，但从工程视角看，它更像是一种 anti-pattern。

从心智模型上，Agentic Coding 和 Vibe Coding 至少有三点区别：

- 是否强调理解：Agentic Coding 默认开发者要理解问题、理解代码和变更影响；Vibe Coding 更强调“先让它写出来再说”，甚至默认可以不理解只要能跑。
- 是否强调验证：Agentic Coding 把验证当作任务的一部分，要求每次变更都尽量通过测试、build 或其他手段确认；Vibe Coding 往往停留在“跑一下、没报错就算过”。
- 是否强调责任：Agentic Coding 明确开发者仍然对结果负责，Agent 只是执行协作者；Vibe Coding 容易让人把责任迁移给模型或工具。

理解这两个词的差异有助于校准心态：Agentic Coding 并不是鼓励你“更大胆地交给 AI 写”，而是鼓励你设计出一套有护栏、有反馈、有验证、有责任的工程方法，把 Agent 当作高效协作者，而不是替罪羊。
