# Part 5 Claude Code 深潜

这一部分会比其他工具写得更深，因为它承担了整本教程的主线角色。原因不是它一定在所有维度都绝对领先，而是它非常适合用来解释 Agentic Coding 的核心机制：终端、仓库、计划、权限、验证、回报。

## Chapter 17 Claude Code 的工作模式

如果用一句话概括 `Claude Code`，我会说它更像“终端里的工程代理”，而不是“聊天框里的代码助手”。

它的强项主要来自三个方面。

第一，它非常强调真实开发环境。不是给你一个脱离仓库的回答，而是让模型站在项目目录里理解代码、读取文件、运行命令、观察结果。

第二，它把任务循环暴露得比较清楚。你很容易看见它什么时候在分析、什么时候在调用工具、什么时候在等待批准、什么时候在输出总结。这对建立 Agent 心智模型非常有帮助。

第三，它天然适合和工程习惯结合。终端用户本来就习惯：

- 在仓库根目录工作
- 用命令驱动 build、test、lint
- 明确知道当前分支和当前工作目录
- 通过文件和命令来理解系统

### CLI 和 IDE 集成怎么取舍

如果你要写教程，最好明确表达一个观点：CLI 和 IDE 不是替代关系，而是主副手关系。

CLI 更适合：

- 陌生仓库探索
- 跨多个文件和目录的操作
- 跑命令、读日志、做验证
- 长任务和阶段性回报

IDE 集成更适合：

- 看 diff
- 局部 review
- 细粒度编辑
- 与现有编辑习惯无缝配合

比较成熟的工作流通常是：主线在 CLI，精细检查在 IDE。

### 什么任务最适合 Claude Code

以下任务通常非常适合：

- 接手陌生项目并快速建立结构感
- 修真实 bug
- 做低到中等风险的功能改动
- 跨多个文件的机械重构
- 补测试、补文档、补脚本

以下任务则不建议一上来完全放手：

- 大规模架构迁移
- 高风险数据库或生产变更
- 目标和验收标准还没想清楚的需求

### 与传统 IDE Chat 的根本区别

最重要的区别不是“会不会写代码”，而是“是不是围绕任务闭环”。IDE Chat 常常停在建议层，而 Claude Code 的典型使用方式应该是：

1. 你给出目标和边界。
2. 它先理解仓库和任务。
3. 它用工具执行。
4. 它根据反馈继续迭代。
5. 它交付结果和验证说明。

如果你把它只是当成聊天面板来用，那你其实浪费了它最有价值的部分。

### Claude Code = 给 Agent 一台计算机

从 xhs notes 的视角看，Claude Code 最值得强调的一点是：它不是在“给模型多几个 API”，而是在“给一个 Code Agent 一台标准化的计算机”：

- 它有一个明确的当前工作目录（当前仓库）；  
- 它可以列目录、读文件、改文件；  
- 它可以运行命令、查看 stdout/stderr；  
- 它可以在这个环境里读写自己的笔记、日志和中间产物。

和很多“纯 API 式 Agent”相比，这种设计有两个显著优势：

1. 行为更可解释：你可以像看一个人类工程师在终端里操作一样，观察它读了哪些文件、跑了什么命令、看到了什么输出。  
2. 能力更可组合：只要某个能力已经有标准 CLI 或脚本实现，Claude Code 就能通过终端调用它，而不需要额外造一个专用工具协议。

这也是为什么本书在讲 Context Engineering 和长任务时，会反复用“把中间结果写进文件系统”“用脚本作为确定性执行单元”这类例子——因为在 Claude Code 这一类环境里，这些做法都非常自然。

## Chapter 18 Claude Code 配置与会话管理

真正开始高频使用以后，问题就不再是“能不能跑起来”，而是“怎么让它长时间保持高质量输出”。

### 先说最稳的配置原则

- 优先使用官方支持路径。
- 把项目级约束写进项目文件，而不是反复口述。
- 保持每次会话的任务边界清晰。
- 不要把所有事情都塞进同一个超长会话。

### 后端模型的选择思路

如果你当前使用的是官方支持路径，优先理解官方推荐的接入方式和权限模型。如果你使用兼容层或聚合层接入其他模型，也要明白这时你是在自己承担兼容成本。

在实践中，模型选择通常不是“哪个更强”这么简单，而是：

- 复杂分析和规划是否稳定
- 代码修改是否收敛
- 工具调用是否可靠
- 成本和延迟是否可接受

### 会话中途切换模型的策略

一个很实用的经验是：不要把模型切换当成情绪化操作，而是当成任务调度。

更适合切模型的时机包括：

- 任务从探索阶段进入批量执行阶段
- 任务从实现阶段进入 review 阶段
- 当前模型反复在同一类问题上失误
- 成本已经不值得继续用高价模型试错

### 权限设置怎么想

权限不是越大越好。更好的原则是：

- 低风险、探索性任务：权限可以略宽，追求效率
- 高风险、生产相关、涉及 secrets 的任务：必须收紧，并保留审批
- 不熟悉的仓库：先收紧，等理解后再放宽

### `CLAUDE.md` 的角色

官方文档里的 memory 机制给了一个很强的实践入口：把长期稳定、确实应该被反复遵守的内容放进 `CLAUDE.md` 或等价记忆文件。

最适合写进去的是：

- 项目常用命令
- 模块边界
- 禁改目录
- 测试习惯
- 输出偏好

不适合写进去的是：

- 某次任务的临时 workaround
- 某次失败后的偶然经验
- 已经过时的版本说明

### 关于 `effort` 与推理强度

如果你当前版本或当前后端暴露了类似 `effort`、reasoning intensity 这类控制项，不要把它当成“越高越好”的按钮。更稳的做法是：

- 复杂定位、方案比较、架构分析：适当提高
- 机械修改、重复修复、文档整理：降低或保持默认
- 长任务中如果已经明确路径：不必一直使用最高强度

## Chapter 19 Claude Code 的高频实用技巧

这一章建议未来补动图，但初稿先把方法写清楚。

### 技巧 1：先用 `/init` 或等价方式建立项目记忆

如果当前版本提供初始化项目规则的能力，建议在刚接手仓库时就做一次。它的价值不是神奇增强，而是把：

- 项目结构
- 常用命令
- 风险边界
- 团队风格

这些长期信息沉淀下来，减少你在每次会话里重复解释。

### 技巧 2：把 `CLAUDE.md` 控制在高密度信息

最好的记忆文件通常不长。它应该像工程手册首页，而不是公司制度大全。一个过长、过杂、互相矛盾的记忆文件，比没有更糟。

### 技巧 3：长任务必须阶段汇报

对于超过 15 到 20 分钟的任务，不要让它闷头做到底。你应该在 prompt 里明确要求：

- 每完成一个阶段就总结
- 遇到不确定项立即暂停
- 不要自行扩大任务范围

### 技巧 4：控制 stdout

终端型 Agent 非常容易在日志海里迷失。你要主动提醒它：

- 只保留关键错误摘要
- 长输出先截断再总结
- 优先关注失败模块和最后相关片段

### 技巧 5：会话一旦明显污染，就重开

很多人舍不得结束会话，结果越补救越乱。一个判断标准是：如果 Agent 连续两轮都抓着错误前提不放，就应该考虑开新会话，并只带入必要摘要。

### 技巧 6：权限不要常开到最大

很多人为了省一次审批，长期使用最宽权限。短期是省事，长期是高风险。更好的做法是建立两三档常用模式：

- 只读分析
- 可改文件但命令受限
- 可执行更完整动作但保留关键审批

### 技巧 7：学会用历史和复盘工具

像 `Claude Code History Viewer` 这类工具的价值，不是“炫技”，而是帮助你复盘：

- 某次任务为什么成功
- 某次会话从哪一步开始跑偏
- 哪类 prompt 最稳定

如果你真的准备长期使用 Agent，这类复盘工具迟早会有用。

## Chapter 20 Claude Code 生态：Hooks、Subagents、MCP、Skills、插件

很多人学 Claude Code 时，只学会了“直接给任务”。这还远远不够。真正进入进阶阶段后，你要开始理解生态扩展。

### Hooks 适合做什么

Hook 更适合那些“在某个时机自动执行”的动作，比如：

- 在修改后自动跑格式化
- 在提交前自动做检查
- 在生成结果前套用统一后处理

Hook 的价值是把重复性护栏自动化，而不是把复杂业务逻辑都塞进去。

### Subagents 适合做什么

Subagent 更适合把复杂任务拆成角色。典型角色包括：

- Planner：先理解需求并产出计划
- Implementer：执行具体修改
- Reviewer：检查风险和风格
- Verifier：专注验证和回归

不是所有任务都需要多 Agent，但一旦任务变长、变复杂，多角色分工会比单一会话更稳。

### MCP 什么时候比 Skill 更合适

当你要接的是“能力”而不是“经验”时，MCP 往往更合适。比如：

- 浏览器自动化
- 数据库查询
- GitHub 操作
- 外部知识库访问

这类东西本质上是“给 Agent 一只新手”，不是“教它一套新方法”。

### Skill 什么时候更轻量

当你要复用的是“做事方式”而不是“底层能力”时，Skill 更轻量。比如：

- 做架构评审的标准流程
- 写 PR 描述的固定模板
- 开新项目时的初始化清单
- 为某类框架修 bug 的固定套路

### 社区生态怎么用

像 `SuperClaude`、`claude-mem`、`Claude Code History Viewer` 这类项目，很适合拿来吸收思路，但不要生搬硬套。更好的用法是：

1. 看它解决的是什么痛点。
2. 理解它背后的方法论。
3. 把真正适合你项目的部分移植到自己的工作流。

生态的价值从来不只是“装更多插件”，而是让你的 Agent 工作流更可复用、更可治理。

## 进一步阅读

如果你决定把 Claude Code 作为日常主力工具，建议按下面的顺序配合本 Part 阅读官方与社区资料：

- 官方文档  
  - [Claude Code overview](https://docs.anthropic.com/en/docs/claude-code/overview)：对应本 Part 的 Chapter 17，帮助你把“终端里的工程代理”定位和能力边界对上。  
  - [Common workflows](https://docs.anthropic.com/en/docs/claude-code/common-workflows)：对照 Chapter 19 的高频技巧，看官方建议的“读仓库 / 修改 / 验证”流程。  
  - [Memory](https://docs.anthropic.com/en/docs/claude-code/memory)：结合 Chapter 18，具体理解 `CLAUDE.md` 与项目记忆的最佳实践。  
  - [Subagents](https://docs.anthropic.com/en/docs/claude-code/sub-agents) 与 [Hooks](https://docs.anthropic.com/en/docs/claude-code/hooks)：进一步理解本 Part 最后一章里提到的多 Agent 和 Hook 用途。  
  - [MCP](https://docs.anthropic.com/en/docs/claude-code/mcp)：与 Part 6 结合阅读，思考哪些能力应该做成 MCP。  

- 社区生态  
  - [SuperClaude Framework](https://github.com/SuperClaude-Org/SuperClaude_Framework)：关注它如何设计命令体系、计划模板、权限护栏；尝试从中挑 1–2 条规则移植到你自己的 `CLAUDE.md`。  
  - [claude-mem](https://github.com/thedotmack/claude-mem)：用它作为“记忆治理”的灵感来源，思考你自己的项目需要哪些自动或半自动的记忆整理脚本。  
  - [Claude Code History Viewer](https://github.com/MrLesk/Claude-Code-History-Viewer)：配合本 Part 的复盘建议，观察一次完整任务的行为轨迹。  

建议实操顺序：

1. 选一个真实项目，用本 Part 的提示把“读仓库 → 小改动 → 验证 → 总结”完整跑一遍；  
2. 把你在过程中发现的 2–3 条稳定规则填进 `CLAUDE.md`；  
3. 再从上面的官方/社区资料里挑一条你最感兴趣的能力（比如 Subagent 或 Hook），在一个小项目上做一次最小实验，而不是一口气全开。
