# 2026 年代码 Agent 实战手记：Claude Code、Codex、GPT 与 Gemini 的使用体会与组合策略

## 说明

本文主要基于近几个月高强度使用各类 code agent 的一线体感，带有明显的个人样本偏差。这里讨论的不是抽象 benchmark，而是更接近真实工程场景中的"是否省心、是否稳定、是否容易跑偏、是否值得付这个钱"。

## 核心判断：模型智力 ≠ Agent 落地体验

我越来越强烈地感受到，代码场景里真正决定体验的，并不只是模型单轮回答有多聪明，而是下面几件事叠加后的结果：

- 模型本身的智力上限
- 长链条任务中的稳定性
- 工具调用（tool use）是否可靠
- 指令遵循是否稳定
- debug 时是否能快速收敛，而不是在错误方向上越走越远
- 整套 scaffold / agent 框架是否成熟

也就是说，会分析，不等于会执行；会写 plan，不等于会把 plan 稳定落地。很多时候，真正拉开差距的，不是"这个模型比那个模型多会 5 道题"，而是它在真实工程环境里会不会在第 7 步、第 12 步、第 19 步突然脑回路拐进沟里。

## 结论先行：几套组合的主观评价

| 组合 | 定位 | 主要优点 | 主要缺点 | 适合场景 |
|------|------|--------|--------|--------|
| Claude Code + Opus 4.6 | 重型主力、硬仗选手 | 复杂设计、旧代码重构、长链执行都很稳，遇到难题更容易快速收敛 | 贵，是真的贵 | 重要任务、复杂改造、稳定性优先 |
| Claude Code + Sonnet 4.6 | 日常主力、性价比担当 | 遵循 skill、跑日常任务、长任务执行都比较可靠 | 智力上限明显低于 Opus，复杂场景容易力不从心 | 普通开发任务、常规执行 |
| Claude Code + Haiku 4.5 | 便宜快工 | 速度快、便宜，简单任务很合适 | 智力明显不足，不适合复杂任务和长链条任务 | 简单改动、收尾型执行 |
| Codex + GPT-5.3-Codex / GPT-5.4 | 高智商参谋型 agent | 单轮分析深度高、plan 能力强、代码理解能力强 | 执行不够省心，容易跑偏，人工介入频率更高 | 调研、分析、制定 plan、代码审阅 |

从官方定位看，Anthropic 也明确把 Opus 4.6 放在最复杂的分析与编码任务上，把 Sonnet 4.6 作为平衡型主力，把 Haiku 4.5 定位为最快、最省成本的选择；OpenAI 则把 GPT-5.4 定位为面向 agentic、coding、professional workflows 的旗舰模型。

就 API 定价看，我"Opus 省心但烧钱"的感受也不是幻觉：Anthropic 官方文档中，Opus 4.6 / Sonnet 4.6 / Haiku 4.5 的基础输入/输出价格分别约为 $5/$25、$3/$15、$1/$5 每百万 token；而 GPT-5.4 的价格约为 $2.5/$15 每百万 token。钱袋子被 token 啃的声音，是能听见的。

## 1. Claude Code + Opus 4.6：目前最像"能独立干活"的那一个

这是我目前最认可的一套组合。

它能相当好地完成复杂需求设计，也能对庞大、复杂、历史包袱很多的旧代码进行重构。更关键的是，它在执行长链条任务时，长时间不出现明显差错或阻塞的概率，显著高于我用过的其它方案。很多 Codex 会卡住、会绕远路、会在错误方向上反复试错的问题，Opus 4.6 往往能更快收敛。

它还有一定的联网搜索、对齐最新信息与细节的能力，所以在一些涉及依赖、文档、版本差异的任务里，也更像一个能真正"自己推进"的 agent，而不是一个需要你时时刻刻拎着后脖颈纠偏的半自动助手。

截至 2026 年 3 月，就我个人体验而言，Claude Code + Opus 4.6 是当之无愧的最强 code agent + 模型组合。

整体感受就是：省心、可靠、稳定，但烧钱烧得让人心绞痛。

如果用自动驾驶做比喻，它更像 L4 智驾：不是完全不需要人，但绝大多数情况下，它真的能把路开下去，而且开得还比较像样。

Anthropic 官方发布中也强调了 Opus 4.6 在编码、长时 agent 任务、大代码库可靠性、代码审查与 debug 方面的强化，并给了它 1M token 上下文窗口（beta）。这和我在实战里的体感基本一致。

## 2. Claude Code + Sonnet 4.6：最好用的"普通任务执行者"

Sonnet 4.6 的能力上限，明显低于 Opus 4.6，这个差距在复杂设计、复杂重构、复杂 bug 追踪上会很明显。但它价格便宜不少，整体性价比更均衡。

我对 Sonnet 4.6 的评价是：

它在 follow skills 指引与规范、执行日常任务、处理不太需要太多智力的常规工作时，表现相当稳定可靠。之前我也成功用它跑过执行链条很长的任务，说明它并不是"只能做小活"的模型。

但问题也很清楚：一旦问题空间变复杂、需要做很多高质量判断、或者需要在多个候选方案中快速选对路线时，Sonnet 的智力还是会有点不够用。

所以它很适合作为普通任务的默认执行者。任务冗长但不算特别复杂时，用它很舒服；一旦进入真正高难度场景，我通常还是会上 Opus。

Anthropic 官方也把 Sonnet 4.6 定位为最强 Sonnet、覆盖 coding、computer use、长上下文 reasoning 和 agent planning 的平衡型模型。

## 3. Claude Code + Haiku 4.5：便宜、快，但别让它扛太多

Haiku 4.5 是 Claude 系里最便宜的模型，适合简单任务执行，推理速度也快。

但它的问题同样很明显：智力不足。

复杂任务不适合它，长链条任务也不适合它。你让它做点清晰、收敛、边界明确的小活，它很利落；你让它去扛设计、重构、复杂 debug，它大概率就会开始露出"小模型的宿命感"。

所以我的用法很直接：简单任务给 Haiku，普通任务给 Sonnet，硬仗给 Opus。别让拖拉机去跑 F1，这样大家都少受点伤。

Anthropic 官方对 Haiku 4.5 的描述也很接近这个定位：它是目前最快、最省成本的 Claude 模型。

## 4. Codex + GPT-5.3-Codex / GPT-5.4：更像参谋，而不是你最省心的执行官

在代码场景里，Codex 与 GPT-5.4 给我的体感差异并不是特别显著。OpenAI 官方也提到，GPT-5.4 吸收了 GPT-5.3-Codex 的编码能力，同时补上了更强的 computer use 和更长上下文，这多少也解释了为什么二者在很多 coding 场景里的边界不像过去那样泾渭分明。

但就我的实际使用体验来说，OpenAI 这套东西更适合作为**参谋型 Agent**，而不是决策或执行型 Agent。

### 它的优点非常明确：

- 单轮对话里生成的内容深度和质量很高
- 后端 LLM 的分析思考能力很强
- 在代码理解、问题拆解、方案比较、风险识别这类环节，往往会给出很有启发的内容

### 但一旦进入"让 agent 真正去执行任务"的阶段，它就没那么让人省心了。

和 Claude 系模型相比，我明显感觉到：

- 人工参与介入的频率更高
- prompt 意图在多步执行后更容易偏移
- 在一些并不复杂的问题上，也会尝试很多方法但就是没打通
- 工具调用场景下，收敛速度不如 Claude
- 指令遵循体感不如 Claude 稳
- 工程稳定性略逊一筹

我甚至有一种强烈的怀疑：差距未必完全在模型本身，也可能在 post-training、tool use 对齐、以及整个 scaffold 上。

这是我的猜测，不是结论，但体感很强。

如果继续拿自动驾驶类比，它更像是一个模型能力很强的 L2 辅助驾驶：智力不弱，甚至在某些分析场景下更强，但你不能真的放心撒手。它容易出差错，也容易跑偏。

OpenAI 官方对 Codex 的描述其实很积极，强调它能加速从功能开发到复杂重构、测试和代码审查的端到端工程工作；但我的个人样本里，它更像一个很聪明、很能出主意、但执行过程需要盯得更紧的搭档。

## 三个我在工作里实际遇到的例子

下面不是 benchmark，是实打实的踩坑记录。工程世界从不讲武德，最能暴露 agent 真实水平的，往往就是这些不体面的坑。

### 例子一：同一套 skill，Claude 能闭环，GPT-5.4 却在错误方向上越走越远

我有一套写得很清楚的 skills，里面明确写了各种注意事项和细节。

在这套 skills 之下，Sonnet 4.6、Haiku 4.5、GLM5 都多次成功复现过昇腾环境相关安装流程，包括：

- CANN
- torch npu
- drivingsdk

但到了 GPT-5.4 这里，任务却出了问题。

当时它因为 source CANN 包顺序问题，触发了 libhccl.so 报错。

真正让我不满意的不是"它出错了"，而是它出错后的先验假设完全跑偏了：

它没有优先怀疑环境问题，而是直接开始动手改 torch npu 和 drivingsdk 的源码。结果自然是越改越偏，越改越离题，浪费了不少时间和 token，折腾很久之后才回头重新排查环境安装步骤和基础环境配置。

而这个任务本质上只是一个环境配置任务。

在这种场景下，更合理的先验假设应该是：

先怀疑安装顺序、环境变量、依赖版本、基础配置，而不是第一时间冲进源码里大开脑洞。

这个例子给我的启发很大：

**Agent 不是怕犯错，而是怕犯错后没有正确的"第一怀疑对象"。**

Claude 系模型在这种工程常识层面的收敛通常更自然一些。

### 例子二：SSH config 问题，GPT-5.4 反复折腾没搞定，Sonnet 换上来一遍就通

还有一次是 SSH 的 config 配置有问题。

GPT-5.4 为了搞定这个问题，尝试了多种方法，甚至还想跑到 server 端拉监听进程，在 client 和 server 两端反复联调，场面一度相当热闹——热闹得像一只非常聪明但方向感一般的章鱼在机房里同时拧八个扳手。

结果：还是没搞定。

后来我切换到 Claude Sonnet，并且为了避免它被前一个 agent 的失败路径污染，我先删掉了 memory 文件，确保它无法知道 GPT-5.4 已经试错过哪些方案。

在这个前提下，Sonnet 基本是一遍尝试就把问题搞定了。

这个案例让我更确信一件事：

**有时 agent 的关键差距，不在能不能想到很多方法，而在能不能尽快排除那些不该继续试的方法。**

### 例子三：同一个重构项目里，Codex 赢了分析与 plan，Opus 赢了落地与 debug

今年 1 月，我做过一次复杂特性的重构。

当时使用的是那一阶段的 SOTA Opus 4.5 和 Codex 5.2。

在重构前，需要先分析旧实现的问题与潜在 bug。

这一阶段里，我主观上认为 Codex 的分析结果比 Opus 4.5 更深入、更全面。在制定重构 plan 时，Codex 也更胜一筹。

但是，一旦进入"把 plan 真的落到代码里"这个阶段，情况就反过来了：

- Opus 生成代码明显更优
- 实测 bug 更少
- 对 plan 和其它 prompt 指令的遵循更好
- 注释、文档的阅读体验更佳
- 代码更简洁、结构更清晰

而进入真实环境的调试与 debug 时，Opus 也快不少，思路更准。

反过来看，Codex 往往需要更多试错；而且执行链条一长，上下文一臃肿，它会出现明显的"降智感"。

这个项目给我的最终经验非常清晰：

**GPT 用于 plan，Claude 用于执行。**

这不是绝对规律，但至少在我自己的工程样本里，这个分工非常有效。

## 我的实际策略：不要把 Agent 当单体，要把它们当一个协作系统

真正实用的方式，不是押宝某一个"神模型"，而是把不同模型放在它们更擅长的位置上。

### 第一阶段：先调研，不要急着上 Agent 写代码

很多问题和任务，本质上需要的是充分调研，而不是上来就开改。

所以只要遇到不确定、不熟悉、不理解的领域，我通常会先用：

- 网页端 GPT-5.4
- Google Gemini 3.1 Pro
- 更偏学术或更需要系统整理时，用它们各自的 Deep Research 一类能力

Google 官方把 Gemini 3.1 Pro 定位为面向复杂任务的最强模型之一，并在 Gemini app、API、Vertex AI、NotebookLM 等产品中提供；OpenAI 和 Google 也都提供了 Deep Research 这类面向系统调研的能力。

这里一个很现实的点是：

**网页端对话问答、分析、调研报告的成本，通常会比让 Agent 直接进仓库干活低一个数量级。**

原因也很简单：

- 调研类任务通常不需要像代码生成那样输出海量 token
- 输入信息量通常也没有"整个代码仓"那么夸张
- 先把问题摸清楚，后面 Agent 执行时踩坑会少很多

另外，Perplexity 我也相当推荐作为 AI 搜索调研与模型聚合平台；只是它的 Max 计划在我的体感里过于昂贵，所以我暂时还没试。

### 完成调研后，我会这样进入 Agent 执行阶段

#### 普通任务怎么分配

普通任务，我基本就是直接无脑用 Sonnet 4.6。

简单任务，尤其是刚用 Sonnet 或 Opus 跑完一个复杂任务、上下文已经被喂得比较完整的时候，我会直接在 Claude Code 里切到 Haiku 4.5 去接着执行，拿它做便宜且快速的收尾工作。

#### 复杂设计、复杂实现、复杂重构、复杂 bug，我的流程基本是下面这 9 步

**1. 先用 GPT-5.4 吃透陌生代码仓**

如果是陌生代码仓，我会先用性价比更高的 GPT-5.4 去分析代码仓结构、模块关系、调用链和设计意图。

通过持续对话，把这个仓库先"读懂"。

这一步的目的不是让 GPT 直接写，而是先让它成为我的认知放大器。

**2. 先把问题空间收束，再让模型发力**

我会先自己想清楚这些问题：

- 到底要解决什么问题
- 哪些需求是必须满足的
- 理想效果是什么
- 大致思路是什么
- 注意事项有哪些
- 场景约束有哪些

我非常看重约束信息。

因为约束本质上是在帮模型缩小 problem & solution space，让它把智力集中在更收束的区域里，而不是到处发散。

很多人会觉得"给模型更多自由，它会更有创造力"；但在工程里，过大的自由度通常只会带来更离谱的解法搜索空间，任务难度和解法数量会非线性膨胀。

**Agent 表现好不好，很大程度上取决于人类提供的信息质量。**

我通常会带着这些思考继续和 GPT-5.4 对话，最后让它产出一套初始 plan。

**3. 把 GPT 的 plan 交给 Opus 4.6，再让它自己扫描代码仓**

接着，我会带着 GPT 产出的 plan 去和 Claude Opus 4.6 交流，同时也让 Opus 自己去扫描代码仓、理解背景与任务。

这一步确实会浪费一些 token，但我依然觉得很有必要。

因为如果后面真正要让 Opus 或 Sonnet 去执行具体任务，那就必须先把 Claude Code 的 context / memory 尽量预填成高质量状态。

而且 plan 阶段的 token 成本通常还可以接受：

输入虽然不少，但输出一般不会像代码生成那样夸张；纯文档输出通常还没有贵到让人当场灵魂出窍。

**4. 人工审核 plan，再用 Superpowers 把意图对齐到更细**

然后我会人工去阅读和审核这个 plan，并给 Opus 反馈和修改意见。

在这一步，我有时会打开 Superpowers 这个插件（也是一套 skill）。

它会增加 token 消耗，所以我平时默认关闭，只在复杂、长链条、难场景下才打开。

它的价值在于：

- 它会主动引导 Opus 来向我提问，而且问题通常是比较友好的选择题、填空题之类。这能帮我把很多原本模糊的意图、约束、边界条件、想要的最终效果，进一步钉死。
- 有些细节甚至是我自己事先都没想清楚的，反而是在这个交互过程中被逼出来的。

这一步做扎实之后，再进入执行，会稳很多。

**5. 把 plan 变成"带注意事项的执行脚本"**

当 Agent 列出执行步骤之后，我不会就这么把方向盘扔给它。

我会基于这些步骤，在 prompt 里逐步补充：

- 每一步的注意事项
- 我自己踩过的坑
- 我倾向的方法
- 不希望它采用的方案
- 哪些地方需要更保守，哪些地方可以更激进

这一步看似啰嗦，实际上非常有用。

**很多 agent 的失败，不是不会做，而是你没有把"你脑子里那些默认但没说出来的工程经验"显式写给它。**

**6. 让 Opus 4.6 生成代码：半自动优先，全自动按需启用**

真正开始生成代码时，我会视情况决定让它半自动还是全自动。

通常来说，我更推荐半自动。

原因很简单：AI 有概率性失误，半自动更不容易在你没注意的时候悄悄跑偏。

但在效率优先、且我人要去忙别的事情时，我也经常会让 Opus 全自动运行，甚至直接设置成 bypass all permission。

当然，这样是有风险的。更理想的做法一定是：

- 做好环境隔离
- 做好沙箱
- 尽量最小权限
- 保证关键资源有保护

就我个人目前的经历来说，我还没有遇到明显安全事故，这让我体感上觉得 agent 生态整体在变得更成熟、更可靠；但从工程实践上看，这绝不意味着隔离和权限控制可以省掉。

该装护栏的时候，还是得装，别因为最近车没翻就觉得悬崖已经消失了。

**7. 代码生成后，丢给 GPT-5.4 二审，制造两个 Agent 的"GAN 时刻"**

代码生成完成、我自己先过一遍之后，我通常还会把代码再丢给 GPT-5.4 去审阅、挑毛病。

这一步非常有意思。

本质上就是故意让两个 agent 互相拆台，像在工程现场制造一个小型 GAN 时刻：一个负责产出，一个负责挑刺。

实践下来，这种"交叉审稿"非常值。

因为 Claude 和 GPT 往往会盯到不同类型的问题，各自的偏好、盲区和审查角度并不一样。

**8. 再切回 Sonnet 4.6 生成测试用例，进入真实环境调试**

接下来，我通常会把 Claude Code 切回 Sonnet 4.6，让它去帮我生成测试用例。

这一步不一定需要 Opus 级别的高智力，但需要足够稳定、足够便宜、足够听话，Sonnet 往往很合适。

然后在条件允许的情况下，就进入真实开发环境去调试。

**9. 真遇到复杂难解的 bug，再调 Opus 4.6 或 GPT-5.4 上来救火**

最后，如果调试过程中遇到复杂难解的 bug，我会再把 Opus 4.6 或 GPT-5.4 拉回来做问题分析、定位与解决。

这时候具体叫谁上场，取决于问题的性质：

- 更偏执行与快速收敛：我倾向 Opus
- 更偏分析、方案比较、推理解释：我倾向 GPT

## 我现在最稳定的一套经验总结

把这几个月的经验压缩成一句话，大概就是：

**GPT / Gemini 做调研和 plan，Claude 做执行和落地；Sonnet 跑日常，Haiku 跑简单活，Opus 打硬仗；最后再让 GPT 回来挑毛病。**

再压缩一点，就是：

**不要迷信单模型通吃。把 Agent 当团队用，而不是当神谕用。**

这是我目前最强烈的体会。

真正决定上限的，不只是模型有多聪明，而是你是否：

- 先做了足够调研
- 给了高质量约束
- 审过了 plan
- 在关键节点做了人工兜底
- 把不同模型放到了最适合的位置上

这样用下来，Agent 才不只是"能写代码的聊天机器人"，而更像一个开始具备工程协作价值的系统。
