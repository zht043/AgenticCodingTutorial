## 🤖 Assistant

可以把业界从 **LLM → Agent → OpenClaw（更广义上是 Computer Use / 通用电脑操作智能体）** 的演进，概括成 **6 个阶段**。先给你一句话总结：

> **业界先让模型“会说”，再让它“会查”，接着让它“会调用工具”，然后让它“会规划和执行任务”，最后让它“会直接操作软件/电脑”，OpenClaw就是这个“从会说到会干活”的代表性里程碑之一。**[3][4]

---

# 一、总览：从LLM到Agent再到OpenClaw，经历了哪些阶段？

## 阶段0：预训练大模型时代——“会说话的模型”
**时间大致：2020–2022**

这一阶段的核心是：  
- Transformer 大模型能力快速增强  
- 模型擅长问答、写作、总结、翻译、代码补全  
- 但本质上仍是 **“概率生成器”**，主要停留在文本世界里

代表节点：
- GPT-3（2020）
- ChatGPT（2022）把大模型从研究能力变成大众产品入口

**行业意义**：  
这一步证明了 LLM 可以成为新的自然语言交互层，但它仍然只是“回答你”，还不能真正替你完成现实任务。也正因为如此，行业很快发现：**只会对话，不足以形成生产力闭环。**

---

## 阶段1：检索增强阶段——“会查资料”
**时间大致：2022–2023**

这一阶段业界的主要思路是：  
让 LLM 不只依赖训练参数里的旧知识，而是接入外部知识库、搜索、企业文档。  
这就是后来的 **RAG（Retrieval-Augmented Generation）** 路线。

代表能力：
- 文档问答
- 企业知识库问答
- 联网搜索
- 实时信息补充

**为什么会出现这一阶段？**  
因为纯 LLM 有几个痛点：
1. 知识过时
2. 容易幻觉
3. 企业数据不在模型里

所以业界先走了“让模型能查”的路。  
这一步很关键，因为它第一次让模型从“封闭脑袋”变成“能访问外部世界”。[4]

**局限**：  
- 仍然主要是“查完再说”
- 没有行动能力
- 无法完成多步骤任务闭环

---

## 阶段2：工具调用 / 插件阶段——“会用工具”
**时间大致：2023**

这是从 LLM 走向 Agent 的第一个真正分水岭。

代表事件：
- 插件（Plugins）
- Function Calling / Tool Calling
- 各类外部 API 接入
- 代码执行器、浏览器、数据库查询等工具接入

这一阶段的核心变化是：

> 模型第一次不只是“生成答案”，而是可以把用户意图翻译成 **结构化调用**，去使用计算器、搜索、数据库、外部系统。  

这使 AI 开始具备“行动接口”。  
腾讯云的总结也提到，Agent 的核心链路就是：**规划、记忆、工具调度、上下文工程**，其中工具调用是使其从聊天走向执行的关键模块。[4]

**行业为什么往这一步走？**
因为企业和开发者发现：
- 很多任务不是“写一段话”就完成
- 而是要 **查、算、调系统、写库、生成文件、发请求**

于是从“Chatbot”开始过渡到“Agentic System”。[3][4]

**局限**：
- 大多数仍是单步工具调用
- 用户需要强引导
- 任务链条稍长就容易失败

---

## 阶段3：工作流与规划阶段——“会拆任务、会执行流程”
**时间大致：2023–2024**

到这个阶段，业界开始真正讨论 **Agent**，而不只是“带工具的大模型”。

核心技术思想：
- ReAct
- Planning / Task Decomposition
- Reflection / Self-Refine
- Multi-step workflow
- Memory

这一阶段的本质变化是：

> 模型不再只是“看到问题→调一个工具”，而是能把任务拆成多步，再按步骤执行、检查、修正。[4]

比如它会：
1. 先理解目标
2. 规划子任务
3. 调用多个工具
4. 根据结果修正计划
5. 生成最终交付物

这时“Agent”开始形成完整定义。  
腾讯云总结的 Agent = **LLM + Memory + Planning + Tool Use**，就是这个阶段工程认知成熟的体现。[4]

代表现象：
- AutoGPT 热潮
- 各类 Agent 框架
- 企业级工作流编排
- 多智能体协作
- Coding Agent、Data Agent、Research Agent 等垂类出现

**为什么这是产业必经阶段？**
因为单次工具调用还不够，真实工作是链式的：
- 做投研要先搜集 → 筛选 → 分析 → 写报告
- 做运营要先查数 → 对比 → 归因 → 输出建议
- 做编程要先理解需求 → 改代码 → 测试 → 修 bug

**局限**：
- 依赖 API 和预定义工具
- 适合结构化数字世界
- 一旦遇到“没有 API、只有图形界面”的软件，就容易卡住

这就引出下一阶段。

---

## 阶段4：Computer Use / GUI Agent 阶段——“会像人一样操作软件”
**时间大致：2024–2025**

这是从 Agent 到 OpenClaw 这条线里最关键的升级。

核心变化：

> 以前 AI 要干活，前提是系统给它 API；  
> 现在 AI 即使没有 API，也能通过看屏幕、点按钮、输入文字，像人一样使用电脑软件。  

这类能力常被称为：
- Computer Use
- GUI Agent
- Desktop Agent
- Browser Agent
- OS Agent

行业逻辑很直接：  
现实世界里，大量系统 **没有标准 API**，或者 API 权限难拿、集成成本高。  
但它们几乎都有 GUI。  
于是，业界开始尝试让 AI “直接操作界面”。[3][4]

在虎嗅这篇调研中也提到，Anthropic 的 Computer Use 被定义为让 Claude 能“像人一样操作计算机”，这被看作 Agent 从工具调用走向现实执行的重要节点。[3]

这一阶段意味着：
- AI 不再只在“数字接口层”工作
- 而是进入“软件操作层”
- 可以跨系统、跨软件完成任务

**局限**：
- 稳定性仍然不足
- 容易被 UI 变化干扰
- 速度、成本和可靠性还不够工业级
- 安全风险显著上升

而 OpenClaw 正是在这一阶段爆火的。

---

## 阶段5：OpenClaw阶段——“从建议型AI到可执行型AI”
**时间大致：2025–2026（你问题里的语境）**

结合搜索结果里对 OpenClaw 的描述，它被业界看作一个象征性拐点：

> 它推动 AI 从“给建议的助手”变成“真正替你执行任务的代理系统”。[1]

从资料看，OpenClaw 被反复描述为：
- 开源智能体系统
- 个人/桌面级 AI 助手
- 可以在本地或私有环境运行
- 重点不只是对话，而是 **执行**
- 被视为“AI 智能体的操作系统”式架构[1]

它代表的不是单个模型能力，而是一种 **产品范式变化**：

### OpenClaw意味着什么？
1. **从云端建议 → 本地执行**
2. **从API优先 → GUI/系统权限优先**
3. **从“你来做，AI辅助” → “AI来做，你来监督”**
4. **从模型竞赛 → 应用/编排/执行环境竞赛**

搜索结果中有一句总结很准确：  
OpenClaw 的突破，在于“把智能体视作基础设施问题，在模型之外构建结构化执行环境”。[1]

这其实说明业界已经意识到：
- 真正的 Agent 不只是更强的模型
- 更关键的是 **执行环境、编排层、权限、安全、记忆、状态管理**

# 二、为什么业界会从“LLM”走向“Agent”，再走向“OpenClaw”？

可以归纳成 5 个驱动力：

## 1）用户需求从“问答”升级为“代办”
ChatGPT 时代，用户主要问：
- 帮我写
- 帮我解释
- 帮我总结

但很快用户开始想要：
- 帮我订票
- 帮我做表格
- 帮我写报告并发邮件
- 帮我跑代码并修 bug
- 帮我在系统里完成流程

也就是说，需求从 **信息获取** 转向 **任务完成**。[3]

---

## 2）企业要的不是“好玩”，而是ROI
企业真正愿意付费的是：
- 节省人力
- 提高流程效率
- 替代重复劳动
- 形成业务闭环

所以行业从“更会聊天”转向“更会执行”。  
这也是为什么近年来大量企业级 AI 项目都落在：
- 研发 Agent
- 客服 Agent
- 运维 Agent
- 数据分析 Agent
- 办公 Agent

---

## 3）API 世界不够大，GUI 世界才是真实世界
很多系统没有完美 API，或者：
- 接入成本高
- 权限难协调
- 维护复杂

但图形界面几乎无处不在。  
所以从工具调用进一步走到 Computer Use，是非常自然的一步。[3]

---

## 4）模型性能提升后，瓶颈转向“系统工程”
在早期，大家比的是模型会不会答。  
到 Agent 阶段，瓶颈逐渐变成：
- 上下文工程
- 规划质量
- 工具调用正确率
- 状态管理
- 长期记忆
- 安全与权限治理

也就是说，竞争重心从 **模型能力** 转到 **智能体系统能力**。[4]

---

## 5）开源推动了 Agent 的普及和“个人化”
OpenClaw 一类开源项目火起来，一个重要原因是：
- 开发者可自己部署
- 可在本地运行
- 可围绕自身工作流定制
- 不完全依赖大厂闭源平台

这让 Agent 从“企业采购品”开始变成“开发者可组装系统”。

---

# 三、每个阶段的局限，以及为什么会进入下一阶段

## 阶段0（纯LLM）的局限
- 不联网
- 不知道最新信息
- 幻觉明显
- 不能行动

**驱动下一阶段：** 需要接入外部知识 → RAG

---

## 阶段1（RAG）的局限
- 会查不会做
- 只能回答，不能闭环完成任务

**驱动下一阶段：** 需要调用工具 → Function Calling / Plugins

---

## 阶段2（工具调用）的局限
- 工具调用往往是单步的
- 多步骤任务容易失败
- 缺少长期状态和计划

**驱动下一阶段：** 需要任务拆解、规划、反思 → Agent Workflow

---

## 阶段3（工作流Agent）的局限
- 强依赖预定义工具/API
- 遇到 GUI 软件能力不足
- 现实系统互通性差

**驱动下一阶段：** 需要直接操作界面 → Computer Use / GUI Agent

---

## 阶段4（Computer Use）的局限
- 稳定性和鲁棒性不足
- 操作成本高
- 安全风险高
- 容易因界面改版失效

**驱动下一阶段：** 需要更成熟的执行环境与代理系统架构 → OpenClaw / Agent OS 化

---

# 四、OpenClaw在这条演进中的位置，究竟代表什么？

如果不把 OpenClaw 只看作某个单独产品，而看成一个 **阶段符号**，它代表的是：

## 1. Agent从“工作流逻辑”走向“操作系统逻辑”
以前大家做 Agent，更多是：
- prompt
- tools
- workflows

OpenClaw 让大家开始思考：
- 执行环境怎么做
- 权限怎么管理
- 状态怎么保存
- 本地/私有部署怎么做
- 多通道消息与工具怎么接
- 如何让智能体长期运行

这说明 Agent 正在“OS化”。[1]

## 2. Agent从“云端服务”转向“个人智能体”
OpenClaw被多次描述为：
- 个人助理
- 桌面级
- 本地优先
- 私有环境可部署

这意味着 AI Agent 开始逼近“个人操作系统助理”的形态。

## 3. Agent从“辅助决策”转向“代理执行”
这也是最关键的一点：
- LLM：告诉你该做什么
- Agent：帮你调用工具做什么
- OpenClaw：直接替你在软件环境里干活

# 五、如果压缩成一条行业演进主线，可以这样理解

### 第一阶段：**会说**
大模型能自然语言交互，ChatGPT 让 AI 成为通用入口。

### 第二阶段：**会查**
RAG 和联网搜索让 AI 开始接入外部知识。

### 第三阶段：**会用工具**
函数调用、插件、API 调度让 AI 具备初步行动能力。

### 第四阶段：**会规划**
ReAct、工作流、多智能体让 AI 开始能拆任务、做闭环。

### 第五阶段：**会操作电脑**
Computer Use、GUI Agent 让 AI 从“调用API”扩展到“使用软件”。

### 第六阶段：**会持续执行**
OpenClaw 代表智能体开始拥有更完整的执行环境、本地化能力和代理式工作方式。[1][3][4]

---

# 六、给你的一个结论版回答

如果你问“从LLM到Agent最后到OpenClaw，业界经过哪些阶段”，最简洁的结论是：

1. **LLM阶段：** 解决“能不能像人一样对话”
2. **RAG阶段：** 解决“能不能访问外部知识”
3. **工具调用阶段：** 解决“能不能真正调用系统和工具”
4. **Agent工作流阶段：** 解决“能不能规划并完成多步骤任务”
5. **Computer Use阶段：** 解决“没有API时，能不能像人一样操作软件”
6. **OpenClaw阶段：** 解决“如何把智能体变成真正长期可运行、可部署、可执行的个人/桌面代理系统”[1][3][4]
