## 评估维度说明

本矩阵从 8 个维度对每类任务进行评估，帮助团队判断"任务是否适合引入 Agent、引入到什么程度"。

| 维度 | 关注点 | 评测方法 |
|------|--------|---------|
| **正确性** | 输出是否满足需求与事实 | 测试通过率、静态检查、人工核对 |
| **可靠性** | 多次运行是否稳定 | 重复运行统计波动 |
| **可解释性** | 能否说明"为什么" | 结构化理由输出 |
| **上下文保持** | 长对话中是否持续理解项目 | 上下文窗口回归测试 |
| **自动化程度** | 能多大程度减少人工搬运 | 端到端耗时、人工操作步数 |
| **集成性** | 能否接入现有工具链 | IDE/CI 集成复杂度 |
| **可测试性** | 输出是否有确定性验证方式 | 自动化验证覆盖率 |
| **风险可控性** | 错误的代价与攻击面 | 权限审计、沙箱、红队测试 |

评级图例：🟩 优秀 · 🟨 良好 · 🟧 中等 · 🟥 差

## 任务拆解参考表

| [协作] 提issue/跟踪问题 | 结构化描述问题 | 输入：现象/日志；输出：issue、标签、复现步骤 | 环境信息、优先级 | issue模板、日志链接 |

## 优先级建议

基于上述矩阵，团队引入 Agent 的优先级建议：

### 🟢 第一优先：高自动化、即时 ROI

| 任务 | 共同特征 |
|------|---------|
| 代码补全与样板生成 | 可快速验证 |
| 代码风格统一/格式化 | 低不可逆风险 |
| 单元测试生成 | 产出标准化工件 |
| Git Commit/PR/Issue 文案 | 已有产品化支持 |
| API/代码内文档生成 | 上下文局部 |

### 🔵 第二优先：辅助式人机协作

| 任务 | 核心要求 |
|------|---------|
| 跨文件特性实现 | 工件式 + 测试门槛 |
| 重构与依赖升级 | 分阶段验证 |
| 代码审查 | 补充人类审查，不替代 |
| 需求分析/技术选型 | 建议式，人做最终决策 |

### 🔴 不建议自治（至少现阶段）

| 任务 | 原因 |
|------|------|
| 生产发布与回滚 | 不可逆 + 强权限 |
| 事故响应处置 | 需要实时业务判断 |
| 数据迁移执行 | 数据丢失不可恢复 |
| 安全审计最终裁决 | 需要合规/法律判断 |

```text
角色：你是资深软件工程师。目标：在不破坏现有行为的前提下实现需求，并产出可合并的PR级变更。

- 需求/issue：

```
