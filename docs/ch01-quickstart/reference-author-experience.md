# 附录：作者使用体验与心得

> 这篇不是 benchmark 抄表，而是偏实战体感：**哪些组合真能长期用、哪些组合看起来强但不一定省心**。
>
> 适合已经跑通一个 Agent、正准备决定“主力工作流”的读者。

---

## TL;DR：我现在怎么搭配

| 场景 | 我更常用的组合 | 为什么 |
|------|----------------|--------|
| **重型复杂任务** | `Claude Code + Opus 4.6` | 多文件联动、长链路执行、收敛稳定性最好 |
| **日常主力开发** | `Claude Code + Sonnet 4.6` | 体验稳、成本比 Opus 低，覆盖了大多数真实开发任务 |
| **方案讨论 / 审查 / 后端逻辑** | `Codex CLI + GPT-5.4 Pro` | 擅长 plan、review、风险识别，沙箱和可控性也更强 |
| **新手 / IDE 优先** | `Cursor` | 上手最快，边看 diff 边协作，心理负担最小 |
| **免费试用 / 超长上下文补位** | `Gemini CLI` | 免费额度友好，长上下文很适合探索大仓库 |
| **中文 / 大陆 / 成本敏感** | `Kimi Code + Kimi K2.5`、`OpenCode + DeepSeek-V3.2`、`Claude Code + GLM-5` | 中文体验、网络条件、成本控制上更容易落地 |

## 1. 我对几条主线的真实感受

### `Claude Code + Opus 4.6`

这是我打硬仗时最放心的组合。它最像“能持续推进任务的成熟工程师”，不是只会给你一段看起来很聪明的答案，而是能把读代码、改代码、跑命令、继续修复这条链路真正跑下去。

我最喜欢它的地方有三个：

- **大仓库理解比较稳**：跨文件、跨模块、跨多轮会话时不容易立刻失真。
- **执行闭环能力强**：更愿意把事情做完，而不是停在“思路很好”的阶段。
- **输出更容易 review**：改动摘要、风险描述、下一步动作通常更像一个会协作的同事。

但它也不是完美的：

- **贵**，而且复杂任务下 token 消耗会明显高于便宜模型。
- **有时太认真**，简单任务也可能写得偏长、偏重。
- **会话脏了以后照样会跑偏**，并不是“强模型 = 不需要上下文工程”。

### `Claude Code + Sonnet 4.6`

这是我更常驻的日常主力。很多真实工作并不需要上来就动用 Opus，Sonnet 这一档往往已经足够。

如果用一句话概括：**最像“省心版主力车型”**。

- 日常增量开发、补测试、修小 bug、写文档、做中等规模重构，都很够用。
- 对遵守工作流、照着规则做事这件事，体感上仍然比较可靠。
- 真到极难问题上，智力上限还是会比 Opus 更早露出来。

### `Codex CLI + GPT-5.4 Pro`

我不会把它当成默认唯一主力，但我很愿意把它放进工作流里当“第二大脑”。

它尤其适合这些环节：

- **方案比较**：当我想听更像“架构讨论”的回答时，它通常会给出不错的拆解。
- **后端 / 逻辑 / 风险审查**：在后端逻辑、接口边界、约束条件这类问题上，常有很强的分析味道。
- **审查与对照**：拿来独立 review 另一条实现路线，很有价值。

它的问题也很明显：

- **执行时更容易需要人盯**，尤其是长链路工具调用任务。
- **输出有时很像高级 release notes**，深度是够的，但 review 成本不一定低。
- **单看“智力强”不等于“长期最省心”**，这是我反复验证下来的结论。

### `Cursor`

如果我要给第一次认真用 Agent 写代码的人推荐一个“最不容易被劝退”的入口，`Cursor` 仍然很强。

- 你可以直接在 IDE 里看文件树、看 diff、看编辑结果。
- 交互成本低，适合边学边改、边聊边看。
- 对很多人来说，**不是最强自治工具，但可能是最好的日常上手工具**。

如果你主要是终端流、超长任务、严格流程控制，那我还是更偏向 `Claude Code` / `Codex CLI` 这类 CLI 主线。

### `Gemini CLI`、`Kimi Code`、`OpenCode`、`GLM-5`、`DeepSeek-V3.2`

这几条在我这里更多是“补位路线”：

- `Gemini CLI`：适合做免费试水、长上下文探索、大体量阅读。
- `Kimi Code + Kimi K2.5`：中文友好，整体体验对国内用户更自然。
- `OpenCode + DeepSeek-V3.2`：低成本、多模型、适合做预算敏感型工作流。
- `Claude Code + GLM-5`：如果网络和合规条件受限，这是一条更容易落地的替代路线。

关于 `GLM-5`，这里顺手澄清一句：**更准确的说法是“与昇腾生态已有适配 / 支持”，而不是简单写成“它一定是在昇腾上训练出来的模型”。**

## 2. 为什么我不建议只看榜单

排行榜当然重要，它能帮你先缩小范围；但长期体验并不只由排行榜决定。

我更看重下面这几件事：

- **长任务会不会越做越偏**
- **改完之后是不是容易 review**
- **对指令和约束的遵守是否稳定**
- **出错时是快速收敛，还是开始原地打转**
- **你的钱包能不能扛住**

这也是为什么我不会因为某个模型榜单分数高，就立刻把它放进“主推第一梯队”。

### 关于 `MiniMax`

`MiniMax` 近阶段在不少榜单上表现确实很亮眼，这点应该正视；但如果只看榜单，很容易得出过于乐观的结论。

我更建议读者把它当成一个**必须亲自验证、必须多看社区口碑**的对象，因为：

- 网上真实用户评价明显是**褒贬不一**的。
- 厂商普遍都会对公开 benchmark 做针对性优化，这并不是某一家独有现象。
- 排行榜里的任务形状，和你自己的真实工作流可能差很多。

所以如果你真的想用 `MiniMax-M2.7` 或相邻版本做主力，不要只看排行榜。请至少去这些地方多搜一圈：

- `Google`
- `Reddit`
- `小红书`
- `知乎`
- `B 站`

## 3. 我反复验证下来最有用的几条心得

### 1. 先选一条主线，再谈横向对比

新手最常见的问题不是“工具不够强”，而是同时装太多、每个都半懂不懂。先用一条主线跑通 10 次真实闭环，比横向试玩 6 个工具更重要。

### 2. 复杂任务一定先出 plan

我现在几乎不会让 Agent 上来就直接改。先让它分析、拆解、列风险，再确认执行，整体成功率会高很多。

### 3. 验证比生成更重要

真正让人翻车的，往往不是“它没写出来”，而是“它写得像对的，于是你就信了”。测试、截图、命令输出、diff 审查，这些比花哨提示词更重要。

### 4. 会话管理是核心技能

上下文一脏，再强的模型也会掉智。任务切换、纠错过多、日志过长、规则文件太肥，都会让体验明显下滑。

### 5. 贵模型用在难点，便宜模型用在执行

我越来越倾向于把工作拆开：

- 难的分析、关键设计、复杂重构，用更强模型。
- 日常执行、批量改动、补齐文档，用更便宜或更快的模型。

## 4. 最常见的坑

| 坑 | 为什么危险 | 更稳的做法 |
|----|------------|------------|
| **一上来装很多工具** | 每个都懂一点，最后谁也用不好 | 先固定一条主线，再加一条备线 |
| **不让 Agent 先分析** | 很容易带着错误理解直接开改 | 复杂任务先让它列计划和风险 |
| **只看生成，不看验证** | 看起来合理的代码最危险 | 测试、截图、命令结果必须跟上 |
| **把一个会话用成垃圾桶** | 早期错误会污染后续判断 | 不相关任务及时清上下文或新开会话 |
| **只崇拜排行榜** | 你会误把“榜单好看”当成“长期好用” | 用 benchmark 做筛选，用真实任务做决策 |
