## 1. 为什么很多人会觉得 Agent 好用但不稳

因为很多人真正采用的流程，其实是：

```text
想到需求 -> 直接丢给 Agent -> 看到输出 -> 凭感觉判断
```

这个流程短期看很爽，长期看最容易出问题：

- 任务理解偏了，但没人及时纠偏
- 中间过程不可审查
- 修改后没有外部验证
- 一旦失败，很难定位到底是理解错了、实现错了，还是验证不够

---

## 2. 先探索，再计划，再执行

更稳的最小流程应该是：

1. **Explore**：先读 README、目录、相关文件，理解约束
2. **Plan**：先给方案，再决定改哪里
3. **Act**：按范围执行，不把改动铺得太开
4. **Verify**：用测试、构建、截图或命令输出给证据

对于陌生仓库，探索阶段至少要回答：

- 这个项目用什么技术栈
- 哪几个目录最相关
- 现有模式是什么
- 目标需求最可能落在哪些文件

> 🔍 **探索不是浪费时间，而是在降低后面的误改成本。**

---

## 3. 规划不是形式主义

很多人反感“先出计划”，其实反感的是那种空洞计划。

一份值得要的计划，至少应包含四件事：

| 维度 | 最低要求 |
|---|---|
| Scope | 改哪些文件，不改哪些文件 |
| Order | 先做什么，后做什么 |
| Verify | 改完用什么方式判断有没有做对 |
| Risk | 哪一步最容易翻车，需要额外注意什么 |

如果这四件事说不清楚，Agent 往往就会开始“边做边猜”。

---

## 4. 验证不是最后补一刀

对 Agent 来说，“我改好了”没有意义，**“我改好了，而且这是证据”** 才有意义。

最常见的验证方式有四种：

- 🧪 测试
- 🏗️ 构建 / 类型检查 / lint
- 🖼️ 截图 / 页面行为验证
- 📄 diff + 日志 + 命令输出

推荐你把“验证要求”直接写进任务里，而不是等它改完了再想起来补一句。

例如：

```text
实现完成后请运行相关测试。
如果没有测试，至少运行构建或 lint。
最后告诉我你实际跑了什么，以及结果是什么。
```

---

## 5. 一个能复用的最小闭环

下面这段话，适合成为你大量任务的默认起手式：

```text
请先探索相关代码和文档，理解当前实现。
然后给出一个简短计划，说明会改哪些文件、如何验证。
等我确认后再执行。
修改完成后请运行相关验证，并告诉我你实际跑了什么。
如果发现信息不足或约束冲突，请先停下来说明。
```

如果你只想带走一件事，就带走这一条：

> 🧭 **让 Agent 稳下来的关键，不是更会说，而是把探索、计划和验证放回主流程。**

---

## 📌 本章总结

- 新手最常见的问题，不是不会写 Prompt，而是跳过了 Explore、Plan 和 Verify。
- 计划要能回答范围、顺序、验证和风险。
- 验证要前置写进任务，而不是最后补一句“顺便测一下”。
- `Explore -> Plan -> Act -> Verify` 值得成为你的默认协作骨架。
