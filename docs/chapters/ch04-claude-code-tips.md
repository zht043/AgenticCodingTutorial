## 0. 本章定位

这一章不追求把 Claude Code 的所有功能一次讲全，而是先让你建立一个正确的入门顺序：

1. 先跑通最小闭环
2. 再写好 `CLAUDE.md`
3. 再开始做高频任务
4. 最后再补会话管理、模型切换和扩展能力

> 🎯 对新手来说，最容易走偏的方式不是“不会用”，而是“一上来就装太多、配太多、想太多”。

---

## 1. Claude Code 最小工作流

Claude Code 最值得记住的，不是某个具体命令，而是这条最小闭环：

```text
打开项目 -> 说明目标 -> 先探索 -> 再计划 -> 执行修改 -> 跑验证 -> 看 diff -> 决定是否继续
```

把它压成一句话，就是：

> 🔁 **Explore -> Plan -> Implement -> Verify**

日常第一轮对话，推荐直接带上这三句约束：

```text
1. 先分析再执行，先告诉我你打算怎么做。
2. 修改后必须验证，告诉我你实际跑了什么。
3. 如果不确定，就停下来说明，不要猜。
```

这三句不是“Prompt 技巧”，而是你和 Agent 的最小协作合同。

## 2. 三种权限心智模型

Claude Code 真正需要理解的不是菜单名字，而是“这次任务到底适合放多少权”。

| 模式 | 更像什么 | 适合什么场景 |
|---|---|---|
| 🛡️ 保守模式 | 你逐步放行 | 第一次接手陌生仓库、敏感项目、生产相关任务 |
| ✏️ 自动编辑 + 命令确认 | Agent 可以自由改文件，但运行命令前让你拍板 | 大多数日常开发任务 |
| ⚡ 高自治模式 | 沙箱里的快速执行器 | 隔离环境、原型实验、低风险批量操作 |

最稳的默认建议是：

- 日常开发先从“自动编辑 + 命令确认”开始
- 高风险任务不要偷懒，始终保留人工批准
- 真想全自动，就先给它沙箱或 worktree，而不是直接放开真环境

## 3. CLAUDE.md：先把项目规则写出来

`CLAUDE.md` 的核心价值不是“写一份很长的说明书”，而是把每次都会重复解释的项目事实前移到控制面。

一份够用的 `CLAUDE.md`，至少包含：

- 📌 项目简介
- 🧱 技术栈
- 📂 项目结构
- ▶️ 常用命令
- 📏 编码规范
- ⚠️ Agent 容易踩的坑

可以先从这个骨架开始：

```md
# 项目简介
一句话说明项目是什么。

# 技术栈
- Framework:
- Language:
- Testing:

# 项目结构
- src/
- tests/

# 常用命令
- dev:
- test:
- lint:

# 编码规范
- 约束 1
- 约束 2

# Agent 容易踩的坑
- 不要改完 X 忘了执行 Y
- 某些目录不要动
```

> 💡 重要决策、约束和项目事实，写进文件才叫“记忆”；只留在对话里，只是临时笔记。

## 4. 第一批最值得做的任务

新手最适合先用 Claude Code 做这些任务：

| 任务 | 为什么适合 |
|---|---|
| 🔍 理解陌生仓库 | ROI 很高，容易看出 Agent 是否真的读懂结构 |
| 🐛 修一个边界清楚的 Bug | 有明确验证信号，不容易陷入抽象争论 |
| 🧪 补单测或回归测试 | 成功标准清楚，最容易建立信任 |
| 📝 改 README / 注释 / 变更说明 | 反馈快，风险低 |

最不建议一上来就做的任务：

- 复杂架构改造
- 高风险发布脚本
- CI/CD 和权限体系大改
- 横跨大量模块的大型重构

## 5. 会话管理与模型切换

Claude Code 好不好用，很大程度取决于你会不会管理会话，而不是会不会写很花的提示词。

### 什么时候该压缩或新开会话

- 子任务完成后：压缩为阶段摘要
- 任务切换明显时：直接新开会话
- 同一个错误来回两次以上：不要硬顶，清上下文重来
- 历史越来越长但有效信息越来越少：停止续聊，重开干净上下文

### 什么时候切更强模型

把 Sonnet 理解成默认主力，把 Opus 理解成攻坚模式会更稳：

- Sonnet：日常实现、补测试、文档、一般修复
- Opus：跨文件重构、难调试、复杂依赖、需要更深推理的任务

推荐做法：

1. 先用默认模型推进
2. 连续两轮没进展就停
3. 压缩无关历史
4. 再切更强模型处理真正卡住的问题

## 6. 一个够用的起手式

下面这段起手式，足够应付你第一次认真把 Claude Code 用在项目里：

```text
请先阅读 README 和目录结构，理解这个项目。

然后告诉我：
1. 这个项目的技术栈和核心模块
2. 如果要做 [我的目标]，你建议从哪里入手
3. 先给出计划，等我确认后再修改

修改后请运行相关验证，并告诉我你实际跑了什么。
如果不确定，就停下来说明，不要猜。
```

如果你只记住一条原则，就记这条：

> 🧭 **Claude Code 不是“让它直接写”，而是“先让它读懂，再让它按边界执行”。**

---

## 📌 本章总结

- 先建立最小闭环，不要一上来堆配置和扩展。
- `CLAUDE.md` 是杠杆率最高的文件，先把项目事实写进去。
- 新手优先做高反馈、低风险、可验证的任务。
- 会话管理和权限边界，往往比“提示词漂亮不漂亮”更影响结果。
