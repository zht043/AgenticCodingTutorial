## 1. Hook 是什么

Hook 更像一个事件触发点：

> 在某个流程节点自动插入一段动作。

它不负责提供推理，也不负责标准化连接，而是负责把动作挂到“正确的时机”。

## 2. Hook 适合挂在哪些时机

常见场景包括：

- 提交前检查
- 文件改完后自动格式化
- 会话压缩后写摘要
- 某条命令执行完后同步产物

## 3. Hook 和 Skill / MCP / Command 的区别

- Skill 偏方法
- MCP 偏连接
- Command 偏显式手动入口
- Hook 偏自动化时机

所以 Hook 最适合承担“流程里的机械触发”，而不是承担复杂推理本身。

| 你想解决的问题 | 更适合什么 |
|---|---|
| 某个固定时机自动做一个短动作 | `Hook` |
| 教 Agent 这类任务应该怎么做 | `Skill` |
| 让 Agent 连上 GitHub、数据库、浏览器等外部能力 | `MCP` |
| 让你手动触发一段固定工作流 | `Command` |
| 做复杂判断、比较方案、决定下一步 | Agent 本体 / Planning，而不是 Hook |

## 4. 一个最小可工作的例子

一个很典型的 Hook 场景，是在会话压缩或阶段结束后，自动把摘要写进固定文件。

```text
触发时机：post-compact
动作：把本轮压缩摘要追加到 docs/session-notes.md
目的：让后续新会话也能快速回放阶段结论
```

这里 Hook 的价值不在“替你思考”，而在：

- 时机稳定
- 动作短小
- 可审计
- 不需要每次都靠人手动重复

如果你想把它再压成一句更工程化的判断，可以记成：

> 🪝 **Hook 适合接“固定时机 + 短动作 + 明确副作用”的任务。**

比如：

- `post-compact` 后把摘要写回文件
- `pre-commit` 前跑一次格式化或轻量检查
- 某次生成结束后同步一份任务状态

但它不适合承担“要不要这么做”的复杂判断。

## 5. 什么时候不该用 Hook

这些场景通常不该优先上 Hook：

- 你需要复杂推理或多步比较
- 你其实是在补方法论，应该写成 Skill
- 你需要的是连接外部能力，应该先看 MCP
- 这个动作一旦误触发，代价很高

一个简单判断是：

> ❌ **如果你还得先问“这一步到底该不该做”，那通常还不到用 Hook 的层。**

## 6. Hook 的边界与风险

Hook 的价值很直接，但边界也必须说清：

- 它适合自动触发，不适合代替 Agent 做复杂判断
- 它适合挂机械动作，不适合变成一套偷偷生长的隐式工作流
- 它往往能执行 shell 或调用外部能力，所以也会直接带来安全与审计问题

最稳的实践通常是：

- 只给 Hook 明确、短小、可审计的动作
- 让高风险动作继续回到人工确认或验证链
- 把 Hook 当成流程插槽，而不是黑盒自动化

## 7. 一个安装前检查清单

在真正挂 Hook 之前，先过一遍这四个问题：

- 这个触发时机是不是稳定、可预期？
- 这个动作是不是短小、可回滚、可审计？
- 如果它执行失败，会不会污染主流程？
- 这件事是不是其实更应该放进 Skill、MCP 或主工作流里？

如果四个问题里有两个回答不清楚，先别挂。

## 📌 本章总结

- Hook 是事件自动化插槽，不是推理引擎。
- 它最适合“固定时机 + 短动作 + 明确副作用”的任务。
- 能写成 Hook 的动作，最好同时满足可审计、可回滚、低风险。
- 一旦动作开始涉及复杂判断、高风险副作用或外部能力接入，就该回到 Skill、MCP 或主工作流里重新设计。
