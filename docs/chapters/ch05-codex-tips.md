## 0. 本章定位

Codex 更适合被理解成一个**终端里的协作型工程代理**，而不是聊天窗口的另一个皮肤。

它的几个关键特点是：

- 更强调当前工作区与真实文件系统
- 更强调规则文件和执行边界
- 更强调“先检查，再动手，再验证”
- 更适合需要控制感、隔离感和审查感的任务

---

## 1. Codex 的默认协作心智

用 Codex 时，最稳的方式不是“把所有需求扔进去”，而是按下面这个顺序推进：

```text
给目标 -> 限范围 -> 读代码 -> 说明计划 -> 执行 -> 验证 -> 汇报结果
```

推荐你默认把任务描述成这种结构：

```text
目标是什么
范围在哪里
不能做什么
完成后怎么验证
遇到不确定时怎么处理
```

这背后的原因很简单：

> 🎯 **终端代理最大的优势不是“会说”，而是“能在明确边界里推进真实工作”。**

---

## 2. AGENTS.md：把规则放到仓库里

如果 `CLAUDE.md` 更像 Claude Code 的项目规则入口，那么在 Codex 体系里，`AGENTS.md` 就扮演类似的控制面角色。

它适合承载：

- 仓库内默认工作约束
- 允许或禁止的验证方式
- Mermaid、构建、测试等特定仓库规则
- 提交和审查前必须满足的条件

一份够用的 `AGENTS.md`，至少应回答这些问题：

- 默认允许做什么，不允许做什么
- 哪些验证要默认跳过，哪些必须执行
- 引用文件和工具时，应该遵循什么格式
- 哪类操作需要停下来先问人

> 📌 对 Codex 来说，`AGENTS.md` 不是文档摆设，而是运行约束的一部分。

---

## 3. 沙箱、审批与隔离执行

Codex 的一大优势，在于它天然适合被放进“有边界的执行环境”里工作。

### 为什么要重视这层边界

因为很多高风险问题，不是模型太弱，而是：

- 权限过大
- 改动范围不清楚
- 缺乏隔离环境
- 验证和回滚没有前置

### 一个稳妥的默认组合

| 组件 | 作用 |
|---|---|
| 🧱 worktree / 沙箱目录 | 隔离本轮改动，不污染主工作区 |
| 📜 `AGENTS.md` | 把仓库规则写清楚，减少每轮重复解释 |
| ✅ 验证命令 | 让“改完就说完成”变成“有证据地汇报完成” |
| 👤 审批点 | 在高风险动作前保留人工裁决 |

对初学者来说，最容易建立安全感的一点是：

> 🛡️ **让 Codex 在隔离工作树里做事，远比在主工作区里盲目全自动要稳。**

---

## 4. 第一批最适合的任务

Codex 特别适合这些场景：

- 读仓库、看 diff、做结构化总结
- 在清晰范围内改一批文件
- 做规则明确的审查和验证
- 根据 `AGENTS.md` 执行严格流程

不建议新手一上来就让 Codex 做的事：

- 生产环境相关操作
- 发布和回滚
- 权限体系和凭据处理
- 需要大量隐性业务背景的系统级改造

---

## 5. Codex 与 Claude Code 的互补关系

这两者不应该被理解成“谁替代谁”，而更适合被理解成两种不同的主工作流。

| 工具 | 更像什么 | 更适合什么时候用 |
|---|---|---|
| 🟠 Claude Code | 更强的项目协作与长流程工作台 | 探索、规划、多轮协作、方法论驱动的开发 |
| 🟢 Codex | 更强调终端执行边界的工程代理 | 规则明确、重视控制感、偏终端流与隔离执行的工作 |

一个很稳的组合方式是：

1. 用 Claude Code 做探索、拆解和方案讨论
2. 用 Codex 在隔离工作树里执行清晰范围的实现或审查
3. 最后统一回到验证与人工裁决

---

## 📌 本章总结

- Codex 的优势在于终端协作、规则落盘和隔离执行。
- `AGENTS.md` 是 Codex 工作流中的关键控制面文件。
- 对新手最重要的不是“全自动”，而是先建立边界清晰的协作感。
- Codex 和 Claude Code 更像两种互补工作流，而不是简单二选一。
