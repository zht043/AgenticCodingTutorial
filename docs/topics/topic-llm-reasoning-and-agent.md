# 为什么大模型看起来会思考，却还是会随机、会跑偏、会突然变笨？

> 这篇不是想堆术语，而是想先帮你把一件事想明白：
>
> **你面对的不是一个确定执行规则的程序，而是一个会在当前上下文条件下不断续写的概率模型。**
>
> 只要这个底层心智模型没立住，后面关于 Prompt、Context、Memory、Harness 的很多讨论都会显得像玄学。

## 1. 先把最容易误解的地方说清楚

很多新手第一次接触大模型，都会在两个极端之间摇摆：

- 一会儿觉得它只是“高级搜索”或“查答案机器”
- 一会儿又觉得它已经像人一样稳定理解世界

这两种看法都不够准确。

更接近现实的说法是：

> **LLM 是一个大规模条件概率模型。**

它不是从数据库里把标准答案翻出来，也不是像传统程序那样按固定流程一步一步执行。它真正做的事，是在每一步都根据当前上下文判断：

> **“接下来最可能成立的 token 是什么？”**

这也是为什么它能写文章、能翻译、能解释概念、能补代码、能生成工具调用。因为这些看起来完全不同的任务，在模型内部最后都被还原成了同一件事：

- 给定前文
- 预测后文

一旦你接受这个事实，很多现象就不神秘了：

- 为什么换个说法效果会变
- 为什么同一个问题有时会答出不同版本
- 为什么上下文一脏，模型会迅速跑偏

## 2. LLM 到底在做什么

如果把过程压缩成最小模型，LLM 的工作流大概是这样的：

```mermaid
flowchart LR
    A["当前上下文"] --> B["LLM"]
    B --> C["下一个 token 的概率分布"]
    C --> D["采样 / 选择一个 token"]
    D --> E["把新 token 拼回上下文"]
    E --> B
```

你看到的一整段回答，并不是模型先在脑中想完、再一次性吐出来，而是这样一个循环重复了很多轮之后形成的结果。

这带来一个很重要的认知转变：

- 传统程序：相同输入通常得到相同输出
- LLM：相同输入得到的是一个分布，最终输出只是这个分布里的一条具体路径

所以，大模型“看起来在思考”，并不意味着它内部有一套像传统推理机那样稳定、显式、可验证的规则系统。它更像一个在海量语言材料里学出了强大模式感的生成器。

它能非常聪明，也能非常不稳。

## 3. 为什么模型处理的是 token，而不是“字词句知识点”

模型真正处理的不是你脑中理解的“一个意思”，而是**token 序列**。

token 可以是：

- 一个字
- 半个词
- 一个完整单词
- 一段符号
- 一小截代码

这件事和 Prompt Engineering 强相关，因为模型并不是直接看到“抽象语义”，而是看到一串有顺序、有位置、有邻接关系的 token。

于是下面这些看似只是文风差异的变化，都会真实影响模型内部计算：

- 规则写在前面还是后面
- 例子放在任务前还是任务后
- 要求写成散文还是写成列表
- 是否用分隔符把不同区块隔开

所以，“Prompt 写得更好”并不只是因为你更会说话，而是因为你重新组织了模型真正看到的输入结构。

## 4. Transformer 为什么会成为主流底座

大家总反复提 Transformer，不是因为它是某种历史名词，而是因为它决定了模型处理序列的方式。

对初学者来说，最值得抓住的是 attention 这一直觉：

> **模型在处理当前位置时，会动态决定自己更该关注前文中的哪些位置。**

这件事带来了两个结果。

第一，模型不再只能顺着一条固定链路往前传信息，它可以更灵活地“回看”前文。

第二，信息之间会发生竞争。也就是说：

- 哪段信息更靠前
- 哪段信息是否被格式包裹
- 哪段信息是否被大量噪声淹没
- 哪段信息离当前任务太远

这些都会直接影响模型到底能不能把那条信息用起来。

这也是为什么后面讲 Prompt 和 Context 时，重点不是“措辞好不好听”，而是**注意力怎么被引导、证据怎么被摆放**。

## 5. 为什么同一个问题会答得不一样

因为模型输出的不是“唯一正确答案”，而是**下一个 token 的概率分布**。

只要系统不是每一步都严格取最高概率 token，而是允许一定采样空间，不同运行就可能在前几步走到不同分支，后面的整段文字也会一起分叉。

最常见的几个参数，本质上都在调这件事：

| 参数 | 它在影响什么 | 直觉理解 |
|------|--------------|----------|
| `temperature` | 分布尖锐程度 | 越高越发散，越低越收敛 |
| `top-p` | 允许参与采样的概率核心区 | 先截掉长尾，再从保留下来的候选里选 |
| `seed` | 随机过程的初始条件 | 有助于复现，但不是严格确定性保证 |

这里最容易误解的一点是：

> **这些参数调的是随机性阀门，不是在凭空提升模型能力。**

把 `temperature` 从 `0.8` 降到 `0.2`，不会让模型突然“更懂”，只是让它更倾向于走高概率、保守、稳定的路径。

## 6. 随机性到底是问题，还是能力的一部分

随机性既是风险，也是能力来源的一部分。

当你让模型做这些事时，适度随机性通常是有益的：

- 头脑风暴
- 提出多个候选方案
- 生成创意文案
- 探索不同写法

但当任务变成下面这些，过高随机性就会迅速转化成故障源：

- 工具调用
- 结构化输出
- 命令参数生成
- 数据库查询条件
- 自动化操作

所以，更成熟的说法不是“Agent 要消灭随机性”，而是：

> **Agent 要重新分配随机性。**

系统会把随机性更多留给探索和候选搜索，而把执行动作、参数格式、状态迁移这些环节尽量收窄。

## 7. 什么叫“推理”：CoT、reasoning model、服务端推理不是一回事

“推理”这个词经常被混着用，但至少可以拆成三层：

### 7.1 Prompt 层的推理诱导

最典型的就是 Chain-of-Thought（CoT）。

它的核心不是让模型“变成另一个模型”，而是通过提示，让模型更愿意把中间步骤展开，从而走上一条更适合复杂求解的生成路径。

所以 `Let's think step by step` 这类提示有时有效，不是因为那句话有魔法，而是因为它改变了输入条件，诱导模型采用不同的求解方式。

### 7.2 模型层的 reasoning model

现在常说的 reasoning model，不只是“你让它写出思路”，而是模型和服务层本身就支持先进行一段内部思考，再给出更凝练的答案。

这和传统 CoT 的区别在于：

- CoT 往往把中间过程直接暴露在输出里
- reasoning model 则可能在内部消耗思考预算，再只给你结果或摘要

### 7.3 服务层的推理优化

同一个模型，服务接口层也可能给你额外的 reasoning 配置，例如：

- reasoning effort
- 内部思考预算
- reasoning summary

这时候，“推理”已经不只是 prompt 技巧，而是一种产品能力。

所以别把这三件事混成一句“让模型多想想”。它们分别对应不同层级的控制手段。

---

## 8. 为什么“让它多想想”有时反而更差

一个很实用的工程判断是：

> **“推理更多”不是目标，“更匹配任务结构”才是目标。**

如果任务只是：

- 提取字段
- 做分类判断
- 填一个 schema
- 调用一个明确工具

那额外的大段显式推理往往只是噪声，既拖慢输出，也可能干扰动作选择。

只有当任务真的涉及下面这些内容时，更多推理才通常有价值：

- 多步规划
- 难题分析
- 冲突约束权衡
- 长链决策

所以，一个成熟系统不会对所有请求都套同一种“深度思考模板”，而会按任务类型切换模型、prompt 和约束强度。

---

## 9. 对 Agent 工程最重要的启发

如果你把前面内容都串起来，会得到几个非常关键的工程结论。

### 9.1 你面对的是概率系统，不是确定系统

这意味着你不能只期待“一次回答就完美正确”，而应该设计：

- 约束
- 验证
- 重试
- 回环反馈

### 9.2 输入结构会改写模型的计算轨迹

Prompt 不是礼貌话术，而是运行时接口。

你怎么组织输入，决定了模型更容易沿哪条路径生成。

### 9.3 上下文质量比“上下文长度”更重要

模型不是越看越清楚。噪声越多，注意力越分散，跑偏概率越高。

### 9.4 Agent 的价值不是“更会说”，而是“更会被系统收束”

因为模型天然带随机性，所以真正决定上限的，最后不是一句 prompt，而是整个 Harness 怎么把它包进闭环里。

---

## 极简记忆版

- **LLM 不是查表器，而是条件概率生成器。**
- **它每一步都在预测下一个 token，而不是一次性想完再输出。**
- **Prompt 能生效，不只是因为“说清楚了”，更因为它改了输入结构。**
- **随机性不是 bug，而是系统原生属性；工程要做的是管理它。**
- **CoT、reasoning model、服务端 reasoning 是三层不同东西，不要混成一句“多想想”。**
