## 模型切换速查

> 在 Claude Code 中，把 Sonnet 4.6 想象成"极其敏捷的高级开发工程师"，把 Opus 4.6 想象成"深思熟虑的首席架构师"。看菜下饭地切换模型，既能保住开发速度，又能在关键时刻调动最强的推理能力。

### 切换方式

**会话内热切换（斜杠命令）：**

```
/model
```

在对话框中输入 `/model` 并回车，终端会弹出交互式列表，用方向键在 Sonnet 和 Opus 之间选择。

**启动时指定模型（命令行参数）：**

```bash
claude --model claude-opus-4-6
```

如果任务一开始就已知极其复杂，可以在启动时直接通过 `--model` 参数指定模型。

### Sonnet 4.6 vs Opus 4.6 选用指南

**Sonnet 4.6（默认，承担 80% 的日常工作）**

适用任务：

- 编写常规业务逻辑（增删改查 CRUD）
- 补充单元测试或注释
- 修复报错信息明确的 Bug（类型不匹配、空指针异常等）
- 代码格式化与简单重构

核心优势：快与省。生成速度极快，Token 成本低，Agent 大量读取文件和搜索时不会产生高额费用。

---

**Opus 4.6（保留给硬核攻坚，承担 20% 的复杂任务）**

适用任务：

- 跨文件的全局重构（同时修改数据模型、中间件、前端接口，牵一发而动全身）
- 从零设计新系统（高并发队列、复杂鉴权机制、数据库表结构）
- 解决"玄学 Bug"——Sonnet 尝试两次仍未修复，或涉及多线程死锁、内存泄漏等需要深度逻辑推理的问题

核心优势：深与稳。能处理极其庞大的上下文，推理链条深，不会在复杂的代码依赖中"迷路"。

### 黄金连招：会话中途切换 Opus

遇到 Sonnet 陷入僵局时，按以下步骤操作：

1. **打断**：Sonnet 连续两次跑测试都报错，或开始在同一文件里来回修改（死循环迹象），立刻按 `Ctrl+C` 中断。
2. **清理上下文**：输入 `/compact` 或 `/clear`，清理前面无关的试错记录。（**重要**：Opus 按 Token 计费，不要把无效的调试历史传给它。）
3. **切换模型**：输入 `/model`，切换到 Opus 4.6。
4. **重新定位问题**：告诉它："刚才的思路行不通，报错依然是 [Error Log]，请重新审视 X 模块和 Y 模块的依赖关系，给出根本的解决方案。"

这套组合拳让你既享受 Sonnet 的速度和低价，又保有 Opus 的兜底能力。
