# 主流 Agent 工具横评

> 这篇主要比较“**工作台**”而不是“发动机”。
>
> `Agent` 更像你实际使用的产品形态与执行系统，`模型` 则是背后的底层能力。很多人把这两者混为一谈，结果越选越乱。

---

## 先说结论

| 你的情况 | 第一推荐 | 第二选择 | 一句话理由 |
|----------|----------|----------|------------|
| **终端流、正式工程任务** | `Claude Code` | `Codex CLI` | 最接近成熟的软件工程协作工作流 |
| **IDE 流、想快速上手** | `Cursor` | `Trae` / `Windsurf` | 边看代码边协作，学习成本最低 |
| **GitHub 团队协作优先** | `GitHub Copilot` | `Claude Code` | 与 GitHub、PR、企业流程结合更自然 |
| **中文 / 大陆 / 成本敏感** | `Kimi Code` | `通义灵码` / `Trae` / `OpenCode` | 中文体验、网络条件、预算更容易落地 |
| **开源 / BYOK / 可控** | `OpenCode` / `Cline` / `Aider` | `CodeGeeX` | 可自选模型、自托管或自带 Key，更灵活 |
| **通用任务自动化，不只写代码** | `Perplexity` / `Manus` | `ChatGPT Tasks` / `Kimi Agent` | 这类更偏通用 Agent，不是纯 coding 工具 |

## 怎么看 Agent，而不是只看宣传页

我更建议从这五个维度看产品：

- **形态**：CLI、AI IDE、插件、Web Agent，到底哪种更适合你的工作方式。
- **自治程度**：是更像副驾驶，还是更像能自己推进任务的执行者。
- **控制感**：你能不能方便地看 diff、卡权限、做人工接管。
- **生态适配**：是否和 GitHub、企业工具链、MCP、规则系统、团队流程兼容。
- **成本与网络条件**：订阅 vs 按量、官方直连 vs 中国网络环境、BYOK 是否方便。

## 一、国际主流 Coding Agent

| 工具 | 形态 | 更擅长什么 | 更适合谁 | 需要注意什么 |
|------|------|------------|----------|--------------|
| **Claude Code** | CLI + 插件 + App | 复杂重构、长任务闭环、大仓库理解 | 真正要拿 Agent 干活的开发者 | 成本不低；会话管理和验证不能省 |
| **Cursor** | AI IDE | 日常开发、边写边改、可视化 diff、低门槛协作 | 新手、产品工程师、前后端日常开发 | 更偏 IDE 主工作流；深度自治和可控性不一定是最强 |
| **GitHub Copilot** | 插件 / 平台集成 | 补全、团队协作、GitHub 原生工作流 | 已深度使用 GitHub 的团队 | 自治程度通常不如专门的 coding agent |
| **Codex CLI** | CLI + 插件 + App | 方案拆解、后端逻辑、审查、沙箱执行 | 偏终端流、重视隔离和控制感的用户 | 长链路执行时往往更需要人盯 |
| **Gemini CLI** | CLI | 免费试水、长上下文探索、Google 生态协作 | 想先低成本体验 Agent 的用户 | 免费和长上下文是亮点，但实战闭环稳定性并不总是第一梯队 |
| **Antigravity** | AI IDE / 应用构建环境 | 快速原型、Google 生态、全栈应用构建 | 偏原型、全栈试玩和应用搭建 | 产品形态变化快，别过度依赖单一版本口碑 |


> 💡 **Codex 产品形态说明**：2025 年起，Codex 从传统 CLI 模式演进为**多 Agent 软件工程平台**（macOS 桌面应用形态）。核心新能力：
> - **并行 Agent**：多个 Agent 可在同一代码库的独立 worktree 副本中同时工作，避免冲突
> - **Skills 机制**：封装说明、资源和脚本，构建可复用的工程工作流
> - **Automation**：支持按计划在后台执行任务，无需人工守候
> - **内置沙盒**：权限提升时请求许可，高危操作有隔离保护
>
> 这使 Codex 更像"多 Agent 指挥中心"而非传统补全工具；若你看到的信息仍在描述旧 Codex 模型（代码补全 API），那已是较早期的定位。

## 二、国产主流 Coding Agent

| 工具 | 形态 | 更擅长什么 | 更适合谁 | 需要注意什么 |
|------|------|------------|----------|--------------|
| **通义灵码** | IDE 插件 | 插件式接入、阿里云生态、企业落地 | 已在 VS Code / JetBrains 体系里的开发者 | 更像企业稳健工具，不一定追求最强自治 |
| **Trae** | AI IDE | 中文开发、性价比、类似 AI IDE 主工作流 | 想找 Cursor 替代或补位的用户 | 海外口碑和企业生态成熟度仍在继续建立 |
| **Baidu Comate** | IDE 插件 | 百度生态、企业内研发流程 | 百度系或企业内开发流程用户 | 更偏企业场景和平台整合 |
| **CodeBuddy** | IDE 插件 / 企业协作 | 与企微生态和团队沟通衔接 | 团队协作、企业内知识流转 | 更适合已有腾讯生态的组织 |
| **Kimi Code** | Web / IDE / Agent 工作流 | 中文体验、视觉相关任务、低门槛协作 | 中文开发者、产品原型和前端场景 | 复杂长链路工程任务仍要自己验证 |
| **CodeArts** | 云研发平台 | 企业级“自动驾驶式研发”、华为云生态 | 企业项目、平台治理、流程整合 | 更偏平台化与企业采购路线 |

## 三、开源生态 / 更多工具

> 这一组不一定都属于“开源”，但都常被放进“更灵活、可替代、可自定义”的讨论区间里。

| 工具 | 形态 | 更擅长什么 | 更适合谁 | 需要注意什么 |
|------|------|------------|----------|--------------|
| **OpenCode** | 多形态、多模型接入 | BYOK、多模型切换、低成本工作流 | 想自选模型和供应商的用户 | 文档、生态和产品节奏变化快，需自己多验证 |
| **CodeGeeX** | IDE 插件 / 助手 | 免费体验、代码生成、轻量辅助 | 预算敏感、想先试水的读者 | 更像轻量助手，不一定是强自治主力 |
| **Aider** | CLI | git 原生、多文件修改、终端党 | 喜欢明确 diff 和传统终端流的工程师 | 交互体验更硬核，不是最适合新手的入口 |
| **Cline** | VS Code 插件 | MCP、浏览器、命令执行、强可控 | 想开源、想可视化、想自带 Key 的用户 | 能力很强，但需要自己搭好模型和成本策略 |
| **Windsurf** | AI IDE | 日常开发、Agent 化 IDE 体验 | 想对比 Cursor 的用户 | 依赖产品版本和模型配置，长期稳定性要自己试 |
| **Devin** | 云端自主 Agent | 端到端任务委派、异步跑活 | 希望把标准化任务外包给 Agent 的团队 | 成本、权限边界、交付可控性都要仔细评估 |

### 其他值得关注的工具（按场景）

| 工具 | 形态 | 更擅长什么 | 更适合谁 | 需要注意什么 |
|------|------|------------|----------|-------------|
| **Continue** | VS Code / JetBrains 插件 | 私有化部署、可插模型、企业合规 | 需要私有化或完全可控的团队 | 需自行维护模型接入和更新 |
| **Replit Agent** | Web IDE / Agent 平台 | 从需求到部署一体化、快速原型 | 非专业开发者、原型快速验证 | 重度工程任务的细粒度控制较弱 |
| **Amazon Q Developer** | IDE 插件 / AWS 生态 | AWS 云资源对接、权限治理、企业合规 | 重度 AWS 用户、大企业研发团队 | 强绑定 AWS 生态 |
| **Sourcegraph Cody** | IDE 插件 / 平台 | 大型代码库搜索、上下文理解、知识接入 | 大型代码库导航、企业知识库场景 | 更偏代码理解和搜索，自主执行能力有限 |
| **Goose (Block)** | CLI / 本地 | 本地优先、开源、工程自动化（安装/执行/编辑/测试） | 偏好本地执行和开源可控的工程师 | 生态相对较新，社区仍在建立 |

## 四、通用 Agent / Claw

> 这组不要和 `Claude Code`、`Cursor`、`Codex CLI` 这些 coding agent 直接混排看待。它们能写代码，但定位不是“软件研发工作台”。

| 工具 | 定位 | 更擅长什么 | 适合什么场景 | 为什么不该和 Coding Agent 混为一谈 |
|------|------|------------|--------------|------------------------------------|
| **OpenClaw** | 开源通用 Agent / Claw | 常驻式自动化、通用任务编排 | 研究、试验、Life OS 式自动化 | 权限模型、运行方式、目标形状都不同 |
| **Manus** | 通用自主 Agent | 多步骤通用任务执行、资料搜集、流程代办 | 办公流、研究流、通用自动化 | 不是围绕代码仓库和工程验证构建的 |
| **ChatGPT Tasks** | 任务执行引擎 | 通用任务分配与执行 | 日常任务管理、通用自动化 | 代码仓库级协作不是它的核心卖点 |
| **Perplexity** | 深度研究与信息整合 | 搜索、引用、资料梳理 | 做调研、写综述、比信息 | 更像研究助手，不是工程执行主力 |
| **Kimi Agent** | 通用中文 Agent | 长文档、办公三件套、中文场景 | 办公自动化、中文资料处理 | 代码能力有，但定位更宽泛 |
| **ArkClaw** | 云端企业 Agent | 飞书 / 企业生态自动化 | 企业办公和流程自动化 | 仍然更偏业务流而非 coding flow |

> 💡 **同为 Claw，路线相反**：OpenClaw 本地优先、开源、自带 API Key；Manus 云端托管、订阅制（$39–$200+/月信用额度体系）、零配置即用。2025 年末 Manus 已被 Meta 收购，产品路线存在变化可能。两者用户画像基本不重叠——在意数据隐私和成本可控的开发者更适合 OpenClaw，在意快速交付和零运维的用户更适合 Manus。

## 五、不要把这些东西混成一类

### `Coding Agent` 和 `通用 Agent`

- `Coding Agent` 更像“软件研发工作台”，核心是代码仓库、命令执行、测试验证、diff 审查。
- `通用 Agent / Claw` 更像“任务自动化系统”，核心是网页、文档、搜索、办公流程和跨工具编排。

### `AI IDE` 和 `CLI Agent`

- `AI IDE` 更适合边看边改、边试边学、低门槛上手。
- `CLI Agent` 更适合长任务、批处理、脚本化、严格流程控制。

### `官方产品` 和 `开源 BYOK 工具`

- 官方产品一般在默认体验、闭环能力、稳定性上更强。
- 开源 / BYOK 工具则在灵活度、透明度、可控成本上更有优势。

## 六、怎么选，最不容易后悔

### 如果你是第一次认真用 Agent 写代码

先从 `Cursor` 或 `Claude Code` 开始。前者更容易上手，后者更容易建立“真正能干活”的工程协作感觉。

### 如果你已经有正式开发任务

优先在 `Claude Code`、`Codex CLI`、`Cursor` 三条主线里定一个默认工作台，再视情况补一条备线。

### 如果你在中国大陆，或者很在意中文体验

从 `Kimi Code`、`Trae`、`通义灵码`、`OpenCode` 这一侧更容易起步，网络、语言和成本会更友好。

### 如果你在意开源、自托管、BYOK

先看 `OpenCode`、`Cline`、`Aider`，不要一开始就和官方闭源产品用同一套评价标准比较。

# 作者实战体验与心得

> 这篇不是 benchmark 抄表，而是偏实战体感：**哪些组合真能长期用、哪些组合看起来强但不一定省心**。
>
> 适合已经跑通一个 Agent、正准备决定“主力工作流”的读者。

---

## TL;DR：我现在怎么搭配

| 场景 | 我更常用的组合 | 为什么 |
|------|----------------|--------|
| **重型复杂任务** | `Claude Code + Opus 4.6` | 多文件联动、长链路执行、收敛稳定性最好 |
| **日常主力开发** | `Claude Code + Sonnet 4.6` | 体验稳、成本比 Opus 低，覆盖了大多数真实开发任务 |
| **方案讨论 / 审查 / 后端逻辑** | `Codex CLI + GPT-5.4 Pro` | 擅长 plan、review、风险识别，沙箱和可控性也更强 |
| **新手 / IDE 优先** | `Cursor` | 上手最快，边看 diff 边协作，心理负担最小 |
| **免费试用 / 超长上下文补位** | `Gemini CLI` | 免费额度友好，长上下文很适合探索大仓库 |
| **中文 / 大陆 / 成本敏感** | `Kimi Code + Kimi K2.5`、`OpenCode + DeepSeek-V3.2`、`Claude Code + GLM-5` | 中文体验、网络条件、成本控制上更容易落地 |

## 1. 我对几条主线的真实感受

### `Claude Code + Opus 4.6`

这是我打硬仗时最放心的组合。它最像“能持续推进任务的成熟工程师”，不是只会给你一段看起来很聪明的答案，而是能把读代码、改代码、跑命令、继续修复这条链路真正跑下去。

我最喜欢它的地方有三个：

- **大仓库理解比较稳**：跨文件、跨模块、跨多轮会话时不容易立刻失真。
- **执行闭环能力强**：更愿意把事情做完，而不是停在“思路很好”的阶段。
- **输出更容易 review**：改动摘要、风险描述、下一步动作通常更像一个会协作的同事。

但它也不是完美的：

- **贵**，而且复杂任务下 token 消耗会明显高于便宜模型。
- **有时太认真**，简单任务也可能写得偏长、偏重。
- **会话脏了以后照样会跑偏**，并不是“强模型 = 不需要上下文工程”。

### `Claude Code + Sonnet 4.6`

这是我更常驻的日常主力。很多真实工作并不需要上来就动用 Opus，Sonnet 这一档往往已经足够。

如果用一句话概括：**最像“省心版主力车型”**。

- 日常增量开发、补测试、修小 bug、写文档、做中等规模重构，都很够用。
- 对遵守工作流、照着规则做事这件事，体感上仍然比较可靠。
- 真到极难问题上，智力上限还是会比 Opus 更早露出来。

### `Codex CLI + GPT-5.4 Pro`

我不会把它当成默认唯一主力，但我很愿意把它放进工作流里当“第二大脑”。

它尤其适合这些环节：

- **方案比较**：当我想听更像“架构讨论”的回答时，它通常会给出不错的拆解。
- **后端 / 逻辑 / 风险审查**：在后端逻辑、接口边界、约束条件这类问题上，常有很强的分析味道。
- **审查与对照**：拿来独立 review 另一条实现路线，很有价值。

它的问题也很明显：

- **执行时更容易需要人盯**，尤其是长链路工具调用任务。
- **输出有时很像高级 release notes**，深度是够的，但 review 成本不一定低。
- **单看“智力强”不等于“长期最省心”**，这是我反复验证下来的结论。

### `Cursor`

如果我要给第一次认真用 Agent 写代码的人推荐一个“最不容易被劝退”的入口，`Cursor` 仍然很强。

- 你可以直接在 IDE 里看文件树、看 diff、看编辑结果。
- 交互成本低，适合边学边改、边聊边看。
- 对很多人来说，**不是最强自治工具，但可能是最好的日常上手工具**。

如果你主要是终端流、超长任务、严格流程控制，那我还是更偏向 `Claude Code` / `Codex CLI` 这类 CLI 主线。

### `Gemini CLI`、`Kimi Code`、`OpenCode`、`GLM-5`、`DeepSeek-V3.2`

这几条在我这里更多是“补位路线”：

- `Gemini CLI`：适合做免费试水、长上下文探索、大体量阅读。
- `Kimi Code + Kimi K2.5`：中文友好，整体体验对国内用户更自然。
- `OpenCode + DeepSeek-V3.2`：低成本、多模型、适合做预算敏感型工作流。
- `Claude Code + GLM-5`：如果网络和合规条件受限，这是一条更容易落地的替代路线。

关于 `GLM-5`，这里顺手澄清一句：**更准确的说法是“与昇腾生态已有适配 / 支持”，而不是简单写成“它一定是在昇腾上训练出来的模型”。**

## 2. 为什么我不建议只看榜单

排行榜当然重要，它能帮你先缩小范围；但长期体验并不只由排行榜决定。

我更看重下面这几件事：

- **长任务会不会越做越偏**
- **改完之后是不是容易 review**
- **对指令和约束的遵守是否稳定**
- **出错时是快速收敛，还是开始原地打转**
- **你的钱包能不能扛住**

这也是为什么我不会因为某个模型榜单分数高，就立刻把它放进“主推第一梯队”。

### 关于 `MiniMax`

`MiniMax` 近阶段在不少榜单上表现确实很亮眼，这点应该正视；但如果只看榜单，很容易得出过于乐观的结论。

我更建议读者把它当成一个**必须亲自验证、必须多看社区口碑**的对象，因为：

- 网上真实用户评价明显是**褒贬不一**的。
- 厂商普遍都会对公开 benchmark 做针对性优化，这并不是某一家独有现象。
- 排行榜里的任务形状，和你自己的真实工作流可能差很多。

所以如果你真的想用 `MiniMax-M2.7` 或相邻版本做主力，不要只看排行榜。请至少去这些地方多搜一圈：

- `Google`
- `Reddit`
- `小红书`
- `知乎`
- `B 站`

## 3. 我反复验证下来最有用的几条心得

### 1. 先选一条主线，再谈横向对比

新手最常见的问题不是“工具不够强”，而是同时装太多、每个都半懂不懂。先用一条主线跑通 10 次真实闭环，比横向试玩 6 个工具更重要。

### 2. 复杂任务一定先出 plan

我现在几乎不会让 Agent 上来就直接改。先让它分析、拆解、列风险，再确认执行，整体成功率会高很多。

### 3. 验证比生成更重要

真正让人翻车的，往往不是“它没写出来”，而是“它写得像对的，于是你就信了”。测试、截图、命令输出、diff 审查，这些比花哨提示词更重要。

### 4. 会话管理是核心技能

上下文一脏，再强的模型也会掉智。任务切换、纠错过多、日志过长、规则文件太肥，都会让体验明显下滑。

### 5. 贵模型用在难点，便宜模型用在执行

我越来越倾向于把工作拆开：

- 难的分析、关键设计、复杂重构，用更强模型。
- 日常执行、批量改动、补齐文档，用更便宜或更快的模型。

## 4. 最常见的坑

| 坑 | 为什么危险 | 更稳的做法 |
|----|------------|------------|
| **一上来装很多工具** | 每个都懂一点，最后谁也用不好 | 先固定一条主线，再加一条备线 |
| **不让 Agent 先分析** | 很容易带着错误理解直接开改 | 复杂任务先让它列计划和风险 |
| **只看生成，不看验证** | 看起来合理的代码最危险 | 测试、截图、命令结果必须跟上 |
| **把一个会话用成垃圾桶** | 早期错误会污染后续判断 | 不相关任务及时清上下文或新开会话 |
| **只崇拜排行榜** | 你会误把“榜单好看”当成“长期好用” | 用 benchmark 做筛选，用真实任务做决策 |

> 继续读：如果你想把这些心得变成一套更稳的工作流，接着看 [`附录：上下文工程、验证闭环与常见失败模式`](./topic-context-engineering.md)。
