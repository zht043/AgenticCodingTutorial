# Part 4 方法论

这一部分是全书真正的分水岭。因为到了这里，教程不再只是在介绍产品，而是在教读者如何稳定地把 Agent 用好。很多人会把好体验归因于模型，把坏体验归因于运气；但从长期看，Prompt、Context 和 Verification 才是决定上限的三根主梁。

## Chapter 13 面向 Agent 的 Prompt 技巧

给 Chatbot 写 prompt 和给 Agent 写 prompt，不是一回事。Chatbot prompt 更像“提问艺术”，Agent prompt 更像“任务规格说明”。

一个好用的 Agent prompt，至少应该清楚这六件事：

- 背景是什么
- 目标是什么
- 约束是什么
- 允许它操作哪些范围
- 如何验证完成
- 你希望它最后以什么形式交付

一个可以贯穿全书的通用模板如下：

```text
背景：
这里是一个已有项目，技术栈是 X，当前问题是 Y。

目标：
完成 Z。

约束：
不要改动 A，不要引入 B，不要修改无关文件。

上下文：
优先查看这些文件/目录：...

工作方式：
先分析并给出计划，再执行。
如遇不确定点，先停止并说明。

验证：
完成后运行 ...，并说明验证结果。

交付：
输出改动摘要、涉及文件、验证结果、剩余风险。
```

这类 prompt 的好处在于，它把“猜你的心思”变成了“执行你的规格”。尤其对于复杂任务，结构比辞藻重要。

### Prompt 里最容易缺失的部分

很多人会写目标，却不写边界；会写想做什么，却不写怎么算完成。这会导致 Agent 在任务中途不断自我扩张，比如顺手重构、顺手升级依赖、顺手改风格，最后把一个小任务做成大工程。

### Prompt 的正确粒度

太粗，Agent 会乱跑。太细，Agent 会失去自主性。比较稳的做法是：

- 目标写清楚
- 边界写清楚
- 让它自己决定微观步骤

不要把每一行操作都替它写死，除非任务风险极高。

## Chapter 14 计划先行：让 Agent 先分析，再执行

很多失败并不是因为模型不够强，而是因为它还没看清问题就被要求立刻动手。

“先 plan，再 act”不是仪式感，而是一种损失控制机制。计划能让你在真正修改前先检查三件事：

- 它理解的问题是不是对的
- 它打算改哪里
- 它的验证路径是否靠谱

### 什么任务必须先出计划

以下任务基本都应该强制先 plan：

- 跨多个模块的改动
- 涉及数据库、权限、配置、部署的修改
- 老项目、陌生项目、文档缺失项目
- 任何“错一次成本很高”的任务

### 什么任务可以边做边看

以下任务可以适当放松：

- 批量修格式
- 小范围测试补充
- 明确可逆的机械改动
- 你已经非常熟悉的局部代码操作

### 好计划长什么样

好计划不是“先分析，再修改，再验证”这种废话，而是要具体到：

- 会看哪些文件
- 为什么先看这些
- 预计改动哪些位置
- 用什么命令验证
- 哪些地方不确定

### 多模型组合中的计划角色

这也是为什么强模型常常适合先做规划，而便宜模型适合执行。规划阶段最需要的是正确拆解问题和找准风险点，执行阶段则更需要耐心和低成本试错。

例如：

- 让强模型先产出计划和验收标准
- 再让便宜模型按计划完成重复性修改
- 最后由强模型做 review 或收尾

这不是玄学，而是把成本放在最值钱的位置上。

## Chapter 15 Context Engineering

如果说 prompt 是“任务规格”，那 Context Engineering 就是“给 Agent 准备工作环境”。

新手最常犯的错误是把上下文等同于“聊天历史”。真正的上下文工程包括：

- 仓库规则文件
- 项目 README
- 架构文档
- 设计文档
- 测试入口
- 构建命令
- 历史约束
- 当前任务的临时信息

### 哪些信息适合长期固化

以下内容适合放进长期规则或记忆文件：

- 项目目录结构和模块边界
- 构建、测试、lint 命令
- 代码风格和命名规范
- 常见坑和禁改区域
- 团队约定的 Git / PR / 发布流程

### 哪些信息应该临时注入

以下内容更适合作为一次性任务上下文：

- 当前需求背景
- 临时 workaround
- 某个 issue 的具体讨论
- 某次异常日志
- 某个临时分支上的实验性约束

### 常见规则文件的角色

- `AGENTS.md`：适合放面向 Agent 的项目工作规则
- `CLAUDE.md`：适合 Claude Code 的长期项目与个人约束
- Cursor Rules：适合按目录、文件类型、模式做精细化规则
- Copilot Instructions：适合平台或仓库级别的辅助约束

这些文件的共同原则是：简洁、明确、无冲突。规则写得像公司制度大全，Agent 通常执行得更差。

### Context 不要越堆越厚

很多团队一开始很兴奋，恨不得把所有规范都塞进规则文件。几周之后，Agent 开始频繁忽略重点、执行混乱。原因很简单：上下文本身也是资源。你要的是高信噪比，不是资料堆积。

## Chapter 16 Verification Engineering

很多教程写到这一步会突然失去工程味，只剩一句“最后记得检查一下”。这远远不够。对 Agent 来说，验证不是补充动作，而是任务闭环的一部分。

### 验证为什么比“看起来对”更重要

模型特别擅长生成“像对的东西”。如果没有真实环境反馈，它会高估自己的正确率。真正可靠的 Agent 工作流必须把验证嵌进任务说明里。

### 可用的验证手段

- 单元测试
- 集成测试
- build
- lint
- 类型检查
- 截图对比
- 浏览器控制台检查
- 接口返回与日志比对

### 好的完成条件怎么写

不要写：

“改完后确认没问题。”

要写：

“改完后运行 `npm test -- login` 或等价验证命令，若失败则继续修复直到通过。最后说明改了哪些文件、执行了什么验证、还有哪些未覆盖风险。”

### 失败后怎么继续

当验证失败时，不要只把一整个长日志重新丢回去。更好的方式是：

1. 保留关键错误片段。
2. 要求 Agent 基于这个错误重新定位。
3. 要求它说明前一次判断哪里错了。
4. 再进入下一轮修改。

### 最值得固定下来的一个习惯

每个任务的 prompt 里都显式写出：

- 验证命令
- 完成标准
- 失败后的处理方式

这会显著减少“看起来做完了，实际上没跑过”的假完成。

## Agentic Coding 的工程原则（理论小结）

结合业界对 Agentic Workflows / Agentic Coding 的讨论，可以把这一部分的方法论浓缩成几条工程原则，作为你设计自己工作流时的检查清单：

- 开发者负责制：无论 Agent 多强，最终对变更结果负责的一定是人类开发者。不要把责任外包给模型，把它当成协作工具，而不是决策主体。
- 理解先于修改：在动手改代码之前，Agent 应该先证明自己理解了问题、上下文和约束。规划阶段的质量，决定后面执行阶段的上限。
- 验证内嵌流程：测试、build、lint、截图、日志检查不是“可选附加”，而是工作流的一部分。没有验证的“完成”，都应该被视为未完成。
- 人类在环：关键节点要保留人工判断——例如计划确认、高风险改动审批、生产操作授权等。把人从机械劳动中解放出来，而不是从责任中移除。
- 抽象边界清晰：Agent 不是要替代所有抽象，而是要在清晰的模块边界内工作。让它围绕清楚的 API、模块、服务边界做事，比让它在一团混乱的代码里“自由发挥”稳定得多。
- 信息密度优先：Prompt、Context、Memory 都应该追求高信噪比，而不是堆砌。给 Agent 提供“决定问题的那 20% 信息”，比给它 100% 原始信息常常更有效。

如果你在设计任何一个 Agent 工作流时都能对照这几条做自检，这一部分的方法论就已经真正落地，而不是停留在抽象概念上。
