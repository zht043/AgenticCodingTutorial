# Part 0 引言与趋势

这一部分解决两个问题。第一，为什么 Agentic Coding 不是一阵热闹，而是开发工作流正在发生的结构性变化。第二，为什么很多人明明已经在用 AI，却依然没有真正进入 Agentic Coding 的状态。

## Chapter 1 为什么是 Agentic Coding，而不是“会聊天的 AI”

过去几年，很多开发者已经习惯了这样一种工作方式：遇到问题，去搜，去问 Chatbot，复制一段代码回来，再手工改进，再自己跑测试。这个流程当然比完全不用 AI 快，但它仍然是以“人”为主要执行器。模型只是给你答案，真正的仓库理解、文件修改、命令执行、结果验证，依然由你自己完成。

Agentic Coding 的关键变化在于，模型不再只是回答问题，而是被包进一个可持续执行的任务循环。你不再只是在聊天窗口里问“这个 bug 可能在哪”，而是把目标交给一个能读代码、改文件、跑命令、观察错误、再继续修复的系统。换句话说，Agentic Coding 的核心不是“更聪明地生成代码”，而是“更完整地驱动开发闭环”。

这会直接改变三类工作：

- 低价值重复劳动会大量被压缩，比如批量修 lint、补缺失测试、跨文件机械重构、格式迁移、文档生成。
- 中等复杂度的研发任务会从“人手工串联多个步骤”变成“人给出目标和边界，Agent 负责执行和回报”。
- 高复杂度任务不会被完全自动化，但会被拆成“Agent 先探索和收集证据，人再做关键判断”的协作模式。

这也是为什么“AI 聊天很强”不等于“你已经掌握了 Agentic Coding”。前者解决的是单轮理解和生成，后者解决的是任务分解、环境交互、验证闭环和上下文管理。

古法编程、Chatbot 编程、Agentic Coding 可以这样粗略对照：

| 模式 | 核心动作 | 优势 | 局限 |
| --- | --- | --- | --- |
| 古法编程 | 手工搜索、手工修改、手工验证 | 可控、透明 | 重复劳动多，切换成本高 |
| Chatbot 编程 | 问答、补代码、解释错误 | 上手快，思路扩展快 | 仍靠人串起整个执行链 |
| Agentic Coding | 目标驱动、持续执行、环境反馈 | 能承担完整任务闭环 | 对提示词、上下文和权限治理要求更高 |

当然，不是所有任务都应该交给 Agent。适合 Agent 的任务通常满足三个条件：

- 任务目标可以描述清楚。
- 任务结果可以验证。
- 错误成本可控，或者有人工审核环节。

反过来，不适合一开始就交给 Agent 的任务包括：

- 战略性架构决策还没有定。
- 产品需求本身高度模糊。
- 涉及高风险生产数据、付款、权限变更且没有审批链。
- 你自己都不清楚什么叫“完成”。

本章的结论只有一句：Agentic Coding 不是把 Chatbot 换个壳，而是把“模型”升级成“开发流程中的执行者”。理解这一点，后面的工具选择和方法论才不会跑偏。

## Chapter 2 AI Coding 的发展脉络

理解 Agentic Coding，最好先看它是怎么一步步演化出来的。

第一阶段是 Chatbot。这个阶段的大模型主要承担解释、问答、代码片段生成、报错分析等角色。它很强，但它不知道你的真实代码库，也不会自己去看文件、跑测试、观察命令输出。你问得越准，它答得越准；你问得模糊，它就给你一堆听起来像样但未必适用的建议。

第二阶段是代码补全。以 IDE 补全、函数级补全、行级补全为代表。它降低了写样板代码的成本，也让“把想法快速变成局部代码”变得更顺手。但它仍然只是在你敲键盘时辅助你，不会主动承担完整任务。

第三阶段是 IDE Chat。模型开始“知道一些仓库上下文”，可以被 `@文件`、`@错误`、`@diff` 调起，能解释当前项目的问题，也能在编辑器中直接给出 patch。很多人会把这个阶段误以为就是 Agent，但严格来说，它还缺少稳定的任务循环和环境执行能力。

第四阶段是 Workflow/Chain。以 LangChain 等工程化框架为代表，行业开始认真思考一件事：单次问答不够，得把“调用模型、调用工具、调用外部系统”串起来。这个阶段非常重要，因为它让大家习惯了把大模型嵌入流程，而不是把它当成一个单独的聊天页面。

第五阶段是 Agent。真正的转折点出现了。模型不再只是在一个预设流程里填空，而是可以围绕目标不断采取行动、读取环境反馈、根据结果调整下一步操作。对代码任务来说，这意味着 Agent 可以自己去读仓库、搜索文件、编辑代码、运行命令、查看错误、继续迭代。

第六阶段是 Multi-Agent 和 Cloud Agent。这里出现了两个方向：

- 一个方向是本地或终端中的多 Agent 协作，比如规划 Agent、实现 Agent、Review Agent 各司其职。
- 另一个方向是云端 Agent，在隔离环境中并行处理多个任务，再通过 PR 或结果摘要回到你的主工作流。

如果把这条演化线浓缩成一句话，就是：AI Coding 正在从“生成建议”走向“执行任务”。

这一段历史里，有三个误区值得提前说明。

第一，不要把某个开源框架误认为是全部历史。LangChain 很重要，但它不是 Agentic Coding 的全部；它更像是让“LLM 进入工作流”这件事工程化的一个阶段。

第二，不要把某个产品名当成方法论。今天热的是 A，明天可能变成 B，但“目标、计划、执行、观察、验证”的闭环不会消失。

第三，不要把“能调用工具”误认为“已经是成熟 Agent”。成熟 Agent 的真正门槛不在工具数量，而在上下文治理、恢复能力、验证机制和权限边界。

这里建议后续配一张演化图，横轴是时间，纵轴是自主性。图里重点突出四个维度：

- 是否能看见真实仓库
- 是否能调用外部工具
- 是否能根据反馈连续执行
- 是否能自行验证结果

### Agentic Coding 在行业里的位置

“Agentic” 这个词本身来自更广义的 Agentic Workflows / Agentic Systems 概念，在软件开发领域，它大致和 GitHub、IBM 等机构所说的 “agentic workflows” 一致：把 LLM 放进一个以目标为中心、可感知环境、可调用工具、可自我调整的多步骤流程中，而不是孤立的一次问答。业界比较共识的特征包括：

- 任务通常是多步骤而不是单函数生成。
- 系统会根据环境反馈（文件、日志、测试结果）调整行为。
- 任务执行过程中可以调用多种工具和外部系统。
- 人类在关键节点保持在环角色，而不是完全放权。

Anthropic 在提出 Agentic Coding 这个说法时，强调的也是这几个关键词：目标驱动、可观察、可验证、可恢复；并且特别强调开发者要把 Agent 当成“工程协作者”，而不是“黑箱自动驾驶”。这一点和很多企业在谈 Agentic Workflows 时的设计哲学是一致的：不是追求完全自治，而是追求在有护栏、有反馈、有责任划分的前提下尽可能自动化。

从这个角度看，Agentic Coding 更像是一套工程方法论，而不只是某个产品或某种模型的名字。它要求你重新设计：

- 任务如何被描述和拆解
- 如何把工具和外部系统暴露给 Agent
- 如何组织 Prompt、Context 和 Memory
- 如何把验证嵌入工作流
- 如何在团队层面定义权限和责任边界

后面几章会分别从心智模型、工具选择、方法论和生态扩展四个角度，把这套方法论拆开来看。

## Chapter 3 Agent 和 LLM 分别做什么

很多新手一上来会问：“到底是模型厉害，还是 Agent 厉害？”这个问题本身就不够精确。更准确的问法是：模型负责什么，Agent 又在模型外面补上了什么。

可以先把分工拆开：

| 组件 | 主要职责 |
| --- | --- |
| LLM | 理解目标、推理、生成文本或代码、做局部判断 |
| Agent Harness | 把模型放进任务循环，管理 plan、tool use、回报与恢复 |
| Tools | 访问文件、终端、浏览器、搜索、Git、外部系统 |
| Instructions | 约束行为，注入项目规则、工作流、禁令与偏好 |
| Memory | 保留长期约束或从历史中提炼出的经验 |
| Verification | 把“看起来合理”变成“在真实环境中成立” |

所以真正的 Agent，不是“更大的模型”，而是“模型 + 任务循环 + 工具 + 约束 + 验证”。

一个很实用的理解方式是把 Agent 想成一个不断循环的工程执行器：

`Goal -> Plan -> Act -> Observe -> Verify -> Iterate`

举个最典型的例子。假设任务是“修复登录页验证码偶发失效的问题，并补充回归测试”。一个普通 Chatbot 可能会做这些事：

- 根据你的描述猜测可能原因。
- 给你一段可能的修复代码。
- 建议你去检查缓存、状态同步、接口异常。

而一个真正的 Code Agent 会这样工作：

1. 读取路由、组件、接口和相关测试文件。
2. 给出一个问题定位计划。
3. 搜索验证码状态管理逻辑。
4. 修改一到两个文件。
5. 运行测试或本地命令。
6. 观察失败输出。
7. 根据输出继续修复。
8. 最后总结改动并说明验证结果。

这里 LLM 当然仍然重要。没有足够好的推理能力，Agent 会做出很多愚蠢操作。但如果没有 Agent 层，模型再强也还是要靠你手工把这些动作串起来。

理解这个分工后，你对各种产品的看法就会变得更稳：

- 一个 Agent 产品强，不代表它接任何模型都强。
- 一个模型强，也不代表它在所有 Agent 前端里都表现一样。
- 一个团队真正追求的，不是某个单点指标，而是整条链路的稳定产出。

本章最后给一个简单判断原则：如果一个工具只能给你建议，它更接近“AI 助手”；如果它能在约束内自己走完整个任务闭环，它才更接近“Agent”。
