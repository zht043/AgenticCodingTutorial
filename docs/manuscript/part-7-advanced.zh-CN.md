# Part 7 多模型、多 Agent、成本与安全

走到这一部分，说明读者已经不只是想“把 Agent 用起来”，而是想让它进入长期稳定的工程生产力。这时最重要的已经不是单个 prompt，而是资源调度、组织分工、风险控制和长期治理。

## Chapter 25 多模型组合使用技巧

多模型协作的核心不是炫技，而是分工。不同模型的价值不在于排行榜，而在于它们在不同阶段是否划算。

最常见、也最实用的组合方式有三类。

### 组合一：强模型做规划，便宜模型做执行

这是最值得优先推荐的模式。规划阶段最怕想错方向，执行阶段最怕低效率反复试错。因此：

- 强模型负责理解需求、分析仓库、设计计划、识别风险
- 便宜模型负责批量修改、重复修补、文档整理、机械迁移

### 组合二：强模型做关键评审，快模型做探索

当任务本身可验证、可回滚时，可以让更快或更便宜的模型先探索，再让强模型做最终 review。这在大型重构、批量代码现代化、测试补齐场景尤其常见。

### 组合三：不同 Agent 前端承接不同阶段

例如：

- 用 `Claude Code` 做仓库探索和实现主线
- 用 `Codex` 处理可并行委派的子任务
- 用 `Cursor` 做局部 diff review
- 用 `Cline` 做 MCP 与 provider 试验

这不是为了追求复杂，而是为了让每个工具做自己更擅长的事。

### 一个很重要的提醒

不要把多模型工作流写成“神秘配方”。真正可迁移的是决策原则：

- 任务复杂度
- 可验证性
- 错误成本
- token 预算
- 延迟要求

## Chapter 26 Multi-Agent 和 Parallel Agent

多 Agent 不是单 Agent 的自动升级版。很多时候，它只是把一个混乱系统复制成几个混乱系统。

### 什么情况下值得并行

适合并行的任务通常满足：

- 可以拆成彼此相对独立的子任务
- 子任务之间冲突少
- 汇总标准清楚
- 结果容易比较或合并

例如：

- 多个候选方案探索
- 多模块独立迁移
- 大量测试补齐
- 文档、脚本、示例等周边内容并行产出

### 什么情况下不适合并行

- 多个 Agent 会同时改同一段核心逻辑
- 任务边界本身都没想清楚
- 汇总者没有明确的验收标准

### 一个常见分工模板

- Planner：负责问题拆解和子任务划分
- Worker：各自执行子任务
- Reviewer：统一审查冲突、风格和风险
- Integrator：做最终合并和总结

### 并行的关键不是数量，而是职责

成熟的并行工作流不是“同时开五个窗口”，而是：

- 每个 Agent 的任务边界明确
- 每个 Agent 的输入格式一致
- 汇总口径统一
- 冲突解决机制提前定义

## Chapter 27 Token、长任务与成本控制

Agentic Coding 最大的隐性成本，不只是模型单价，而是上下文和试错链路。

### Token 为什么会失控

典型原因包括：

- 不断累积无用历史
- 贴全量日志
- 让 Agent 在不清晰的任务上反复试错
- 一个会话里混太多任务

### 最有效的节约方式不是省字，而是缩短闭环

比起抠提示词的字数，更有效的是：

- 让计划更清楚
- 让验证更直接
- 让错误反馈更聚焦
- 让会话阶段性总结

### 长任务怎么处理

对于长任务，推荐固定使用以下手法：

- 阶段划分
- 阶段总结
- 中途压缩
- 只携带关键结论进入下一阶段

### stdout 过长的处理方式

不要把完整输出喂给 Agent。更稳的办法是让脚本先帮你做摘要，或者手动保留：

- 失败命令
- 退出码
- 关键错误
- 相关堆栈
- 上下文足够的几行前后文

### 成本控制的底层原则

真正成熟的成本控制不是“永远用便宜模型”，而是：

- 把贵的推理用在高价值节点
- 把便宜的推理用在高频重复节点
- 尽量让每一轮对话都推进状态

## Chapter 28 权限、安全与团队治理

到了团队场景，Agent 的问题就不再是个人效率，而是制度设计。

### 权限为什么是第一优先级

一个能读文件、改文件、跑命令、接外部系统的 Agent，如果没有权限治理，风险远高于单纯聊天模型。你至少要区分：

- 只读
- 可写代码
- 可执行本地命令
- 可访问外部系统
- 可触发高风险动作

### 最小权限原则

默认给最小权限，按任务逐步放宽，是最稳的基本盘。尤其是：

- 生产环境
- secrets
- 数据库写入
- 支付、权限变更、账号操作

这类区域必须极度谨慎。

### 第三方 API 和中转商的信任问题

团队在选择兼容 API、聚合层或中转服务时，不能只看价格。更应该问：

- 日志是否保留
- 数据是否出境
- 请求是否二次转发
- 模型映射是否透明
- 安全事件如何响应

### 团队级治理建议

成熟团队通常会做这些事情：

- 给 Agent 使用制定分级权限策略
- 固化规则文件与 prompt 模板
- 要求高风险任务必须保留验证和审批记录
- 明确哪些仓库、哪些目录、哪些命令属于禁区
- 把 Agent 结果纳入 code review，而不是跳过 review

### 安全的目标不是把 Agent 禁掉

很多团队的第一反应是“风险很大，那就别用”。这通常不是最优解。更好的做法是像对待 CI、脚本、自动化平台一样，对 Agent 建立边界、流程和审计。Agent 真正可怕的不是能力强，而是能力强但没有制度。
