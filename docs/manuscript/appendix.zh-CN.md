## Appendix A 术语表

### LLM

大语言模型。负责理解、推理、生成，但不天然等于完整 Agent。

### Agent

把模型放进任务循环中的系统。它不只回答问题，还会围绕目标持续计划、行动、观察、验证和回报。

### Tool

Agent 可调用的能力，例如读文件、改文件、跑命令、访问浏览器、Git、数据库等。

### MCP

Model Context Protocol。一层标准化协议，用于让 Agent 安全、结构化地访问外部能力和数据源。

### Skill

可复用的经验单元或工作流单元，用来把“该怎么做一类任务”沉淀下来。

### Context Window

模型当前能处理的上下文范围。不是越大越好，更重要的是上下文质量。

### Memory

长期保留的规则、偏好、项目信息或历史总结。应当谨慎使用，避免污染。

### Verification

验证环节。包括测试、build、lint、截图、日志检查等，用来确认结果不仅“像对”，而且“真的成立”。

### Provider

提供模型能力的服务方或接入层。可能是原厂、云厂商托管、聚合层或兼容 API 服务。

```md
# Project Memory

## Project Structure
- `apps/web`: 前端应用
- `apps/api`: 后端服务
- `packages/ui`: 公共组件

## Commands
- dev: `pnpm dev`
- test: `pnpm test`
- lint: `pnpm lint`

## Guardrails
- 不要修改 `infra/`，除非任务明确要求
- 修改业务逻辑后必须运行对应测试
- 长任务要阶段性总结
```

### 一个可参考的 `AGENTS.md` 片段

```md
## Working Rules

- 先分析再执行
- 如果任务跨多个模块，先输出计划
- 修改完成后必须运行验证命令
- 不确定时暂停并说明
- 不要修改无关文件
```

### 一个最小 `SKILL.md` 模板

```md
# Skill: Bug Triage

Use this skill when the user asks to locate and triage a bug in an existing repository.

## Workflow
1. Read the project entry points and relevant module files
2. Summarize likely causes
3. Propose a verification plan
4. Only then edit code

## Output
- suspected root causes
- chosen path
- files to inspect
- verification steps
```

## Appendix F Agentic Coding 设计哲学（进阶解读）

这一节把全书分散提到的理念提炼成一组设计哲学，偏理论一点，方便你在做团队规范、内部分享或设计自己 Agent 系统时作为底层“价值观”参考。

### 1. 目的论优先：从“问题是什么”出发，而不是“模型能做什么”

传统工具选型容易从“这个东西很强”开始往回推场景；Agentic Coding 刻意反过来：先问清楚问题和目标，再决定哪些环节适合交给 Agent 接管，哪些环节必须保留人工判断。

具体体现为：

- prompt 里先写目标与验收，再写工具和细节
- 不追求“全自动”，而是追求“高价值阶段自动 + 高风险节点人工”
- 新增任何自动化前先问：它解决的是哪个具体痛点

### 2. 环境耦合：Agent 必须扎根在真实开发环境中

Agentic Coding 与早期“Chatbot 写代码”的根本区别在于：它不是在真空中讨论代码，而是在仓库、终端、CI、知识库乃至生产环境的上下文中运行。

这带来两条设计上的要求：

- Agent 要能读到真实的仓库状态和运行结果，而不是靠猜
- 你要认真设计它能看到什么、能改什么、能触发什么

否则，所谓 Agent 只是一个更聪明的“代码搜索引擎”，而不是真正的工程执行者。

### 3. 工具中心而非模型中心：模型是“控制器”，工具是“效应器”

在 Agent 系统里，把模型当作“控制器”（controller）会更健康：它负责理解、决策和协调；而真正作用于世界的是工具（效应器）：文件系统、终端、HTTP、数据库、浏览器、MCP server 等。

这意味着：

- 新增能力优先考虑暴露一个清晰的工具接口，而不是往 prompt 里塞更多规则
- 把“能调用哪些工具、以什么权限调用”当成系统的 API 设计，而不是临时拼接
- 工具少而精，比多而散更便于治理与审计

### 4. 分解与委派：结构化任务，而不是依赖“万能一次问答”

Agentic Coding 的内核是“任务循环”，不是“一次性问答”。这天然要求你把任务：

- 垂直分解成阶段：理解、规划、修改、验证、总结
- 水平拆分成子任务：不同模块、不同服务、不同责任域

在理论层面，这跟经典软件工程里的分层架构、模块化设计是一脉相承的：你不是追求一个万能的大进程，而是追求可组合、可替换、可扩展的一组协作单元。

### 5. 验证为一等公民：从“看起来对”走向“实测为准”

Agentic Coding 的设计哲学里，验证不是事后检查，而是工作流的一等公民（first-class）。这和测试驱动、持续集成等工程实践是一条线上的：

- 每个任务都应该有明确的验证路径
- 验证应当尽可能自动化
- 失败应当被视为流程的一部分，而不是“坏运气”

如果一个 Agent 工作流没有显式的验证步骤，那从设计哲学角度看，它就更接近“增强版 Vibe Coding”，而不是成熟的 Agentic Coding。

### 6. 人类在环与责任边界：不是“放权”，而是“分权”

Agentic Coding 不是要把人踢出循环，而是要让人从重复劳动中抽身出来，专注于判断和决策。设计哲学上更接近“分权”而不是“放权”：

- 对低风险、高重复、高可验证的部分尽量自动化
- 对高风险、价值判断、伦理和业务责任相关的部分保留人工审批
- 在团队里明确：Agent 把建议写在 PR 里，人来 review 决定是否合入

这和传统的“自动化流水线 + 人工审批节点”的 DevOps 思路高度一致，只是现在流水线中的“可编程逻辑”被 LLM/Agent 替代或增强了。

### 7. 可观察与可追责：从黑箱到可解释系统

一个良好设计的 Agent 工作流应当是可观察、可追溯的：

- 关键步骤要有清晰的日志或记录：看了什么、改了什么、跑了什么命令、得到什么结果
- 让后来的读者能复现当时的决策链路，而不是只看到“一堆改动”

在理论层面，这延续了可观测性（observability）和审计（auditability）的理念：系统不仅要能运行，还要能被理解和回放。

### 8. 安全与最小授权：能力越大，边界越重要

Agent 与传统脚本最大的不同在于：它可以在较模糊的指令下做出“看似合理”的广泛操作。设计哲学因此必须强调最小授权（least privilege）和边界：

- 默认只给完成当前任务所需的最小权限
- 高危操作（生产写入、权限变更、支付等）必须经过显式确认
- 使用兼容 API / 中转商时，把数据出境、日志留存、限速与风控当成系统设计的一部分

这延伸了经典安全工程中的“防御纵深”和“零信任”思路：不要因为是“AI 在操作”就放松安全标准。

### 9. 上下文作为资源：管理信息，而不是堆砌信息

上下文窗口、记忆文件、规则文档等都是稀缺资源。设计哲学要求你像管理缓存/索引一样管理它们：

- 只保留对当前和未来任务有价值的信息
- 定期清理过期、冲突或冗余的记忆
- 把长期规则和短期背景分离管理

这在本质上仍然是“信息架构设计”问题：只是对象从“给人读的文档”扩展到了“给 Agent 读的上下文”。

### 10. 可演化性：把 Agent 看作系统的一部分，而不是外挂

最后一条哲学是：Agentic Coding 应该被看作软件工程体系的一部分，而不是一个外挂。它会随着架构调整、团队规模变化、业务演进而一起演化：

- 新增模块时要考虑如何让 Agent 理解它们
- 调整开发流程时要考虑 Agent 的位置和作用
- 安全策略变更时要同步更新 Agent 的权限和规则

这让 Agent 从“临时提高效率的小工具”升级为“工程系统中的一等公民”。也只有在这时，你投入在 Agentic Coding 上的设计工作，才会在长期持续产生回报。
